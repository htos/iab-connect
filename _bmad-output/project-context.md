# Project Context for AI Agents

_This file contains critical rules and patterns that AI agents must follow when implementing code in this project. Focus on unobvious details that agents might otherwise miss._

---

## Technology Stack & Versions

- Backend is .NET 10 / ASP.NET Core 10 with nullable reference types, implicit usings, warnings-as-errors, central package management, and latest analyzers. Do not downgrade runtime versions or introduce per-project package versions unless explicitly requested.
- Architecture is a modular monolith with Clean Architecture-style boundaries: `IabConnect.Api`, `IabConnect.Application`, `IabConnect.Domain`, `IabConnect.Infrastructure`. Do not introduce microservices for MVP work.
- Persistence is EF Core 10.0.2 with PostgreSQL via Npgsql EF Core 10.0.0. EF migrations belong in `backend/src/IabConnect.Infrastructure/Migrations`; never make manual schema changes.
- Backend patterns are Minimal API endpoint classes, MediatR 12.4.1 commands/queries, FluentValidation 11.11.0 validators, Serilog, Hangfire, QuestPDF, and Swiss QR bill generation.
- Auth uses Keycloak/OIDC with JWT bearer/OpenID Connect. UI hiding is not security; every protected operation must be authorized in the backend, and sensitive actions need audit/privacy consideration.
- Frontend is Node.js >= 22, Next.js 16 App Router with Turbopack, React 19, TypeScript 5 strict mode, Tailwind CSS 4, next-auth, next-intl, TanStack Query, React Hook Form, and Zod.
- Use executable project files as version source of truth: `package.json`, `.csproj`, `Directory.Packages.props`, and config files override older prose docs when versions disagree.
- Frontend UI should reuse Tailwind, Radix primitives, lucide-react icons, and shared components under `frontend/src/components/ui`. Primary color is `orange-600`; do not use blue primary buttons or links. All UI text must use next-intl translation keys.
- Backend tests use xUnit v3, FluentAssertions, Moq, Testcontainers PostgreSQL, ASP.NET Core MVC Testing, and coverlet. New domain entities/use cases need unit tests; repository behavior needs Testcontainers-backed PostgreSQL integration tests.
- Frontend tests use Vitest, Testing Library, and Playwright. Add or update tests when changing shared UI, forms, auth-dependent flows, or critical navigation.
- Local infrastructure is Docker Compose with PostgreSQL 17, Keycloak 26.5.2, and RustFS/S3-compatible storage. Backend and frontend run in separate terminals during development.

## Critical Implementation Rules

### Language-Specific Rules

- C# code must compile with nullable reference types and warnings-as-errors. Prefer `required`/`init` properties, records for request/response DTOs, sealed classes where inheritance is not intended, and explicit `CancellationToken` parameters on async operations.
- Keep C# project dependencies centrally versioned in `backend/Directory.Packages.props`. Do not add package versions directly to individual `.csproj` files.
- Use async/await end-to-end for I/O. Repository, MediatR handler, endpoint, EF Core, and storage/email APIs should accept and pass through `CancellationToken`.
- Backend DTOs should stay explicit and close to endpoint/application boundaries. Avoid leaking EF entities directly to API responses.
- TypeScript runs in strict mode with path alias `@/*` mapped to `frontend/src/*`. Prefer typed DTOs/API helpers over untyped objects and avoid `any` unless there is a narrow documented reason.
- Frontend components use double quotes, semicolons, 2-space indentation, trailing commas where configured, and Tailwind class sorting via Prettier.
- Client components must declare `"use client"` only when they need client-only behavior such as hooks, browser APIs, auth state, or event handlers.
- All user-visible frontend text must use `next-intl` translation keys. Do not add hardcoded German or English UI strings in components.
- Frontend date display must use `toLocaleDateString("de-CH", { day: "2-digit", month: "2-digit", year: "numeric" })` unless a feature explicitly requires another format.
- Frontend enum values must exactly match backend enum names, including PascalCase values expected by API contracts.

### Framework-Specific Rules

- Backend endpoints are Minimal API endpoint extension classes under `IabConnect.Api/Endpoints`. New endpoint groups should follow the existing `MapXEndpoints(this RouteGroupBuilder group)` pattern with route groups, tags, names, descriptions, and explicit authorization policies.
- Keep business rules out of endpoint handlers and EF entities. Endpoints should orchestrate HTTP concerns, authorization, mapping, MediatR/repository calls, audit logging, and response shaping.
- Use MediatR commands/queries and FluentValidation for application use cases that contain business workflow or validation beyond simple reads. Keep validators in the application layer, not in API handlers.
- Backend authorization must be enforced with policies/permissions in the API/application layer. Frontend role checks are only UX controls and never replace backend authorization.
- Sensitive backend actions such as create/update/delete, status changes, privacy actions, finance operations, and access-denied cases should write audit/security logs using the existing audit services.
- EF Core changes should go through repositories/ApplicationDbContext patterns already present in Infrastructure. Do not put direct database queries in controllers/endpoints unless the existing endpoint already uses that pattern for simple read/statistics behavior.
- Next.js uses the App Router under `frontend/src/app`. Authenticated feature pages must use the standard page layout from `docs/13_frontend_design_standards.md`: `<main className="min-h-[calc(100vh-4rem)] p-4 md:p-8 bg-gray-50">` with an appropriate max-width container.
- Reuse existing navigation/auth/layout components before creating new ones: `MainLayout`, `Header`, `Sidebar`, `PublicHeader`, `PublicFooter`, `useAuth`, and shared UI components under `frontend/src/components/ui`.
- Authenticated list/table pages must include search/filter UI above the data table or card grid and must hide unavailable actions based on permissions/roles.
- For frontend data refresh after mutations, prefer a `refreshKey` state plus `useEffect` with cancellation guards. Do not add inline duplicate `api.get/post` refresh chains inside event handlers.
- Use orange primary actions and links (`orange-600`/`orange-700`) and lucide-react icons where available. Avoid introducing new blue primary actions or manual SVG icons when a shared icon/component exists.
- Public website routes live under `frontend/src/app/public` and use the public layout/header/footer instead of authenticated sidebar layout.

### Testing Rules

- Backend tests are split by layer: `IabConnect.Application.Tests` for domain/use-case/validator behavior, `IabConnect.Infrastructure.Tests` for persistence/integration infrastructure, and `IabConnect.Api.Tests` for API/middleware/host behavior.
- Add or update backend unit tests for new domain entities, value objects, domain events, validators, MediatR handlers, and application services. Use xUnit v3 with FluentAssertions; use Moq only for external boundaries.
- Repository and persistence behavior should be tested in `IabConnect.Infrastructure.Tests` with Testcontainers PostgreSQL when behavior depends on real PostgreSQL or EF Core relational semantics. Do not rely on EF InMemory for repository behavior that needs relational correctness.
- API-level tests can use `Microsoft.AspNetCore.Mvc.Testing`; keep them focused on routing, middleware, auth behavior, serialization, and response contracts rather than duplicating all application-layer tests.
- Finance, privacy, audit, authorization, backup/restore, email sending, document storage, and other sensitive workflows need regression tests when changed.
- Frontend currently has test tooling but few visible test files. Add Vitest/Testing Library tests for shared UI, form validation, API helper behavior, auth-dependent rendering, and stateful components when touched.
- Use Playwright for browser-critical flows such as login-dependent navigation, member/event/finance workflows, public pages, and flows where responsive layout or routing matters.
- Before merging backend changes, run `dotnet test` from `backend`. Before merging frontend changes, run at least `npm run typecheck`, `npm run lint`, and relevant Vitest/Playwright tests from `frontend`.

### Code Quality & Style Rules

- Respect `.editorconfig`: LF line endings, UTF-8, trim trailing whitespace, final newline, 2-space default indentation, 4-space indentation for C#.
- Backend builds enforce analyzers, code style, nullable reference types, and warnings-as-errors. Fix warnings instead of suppressing them unless there is a narrow, documented reason.
- Keep module boundaries clear in the modular monolith. Do not create cross-module shortcuts that bypass Application/Domain abstractions or Infrastructure repositories/services.
- Documentation updates must follow existing `docs/` conventions. Requirements content comes from `docs/Anforderungen_WebApp_Indischer_Kulturverein.csv`; status belongs in `docs/10_requirements_status.md`; do not edit requirement CSV content unless explicitly requested.
- When requirement status changes, update `docs/10_requirements_status.md` and keep `docs/01_requirements.md` consistent as the readable view.
- Frontend formatting is governed by Prettier: double quotes, semicolons, 2-space indentation, print width 80, trailing commas where configured, and Tailwind class sorting.
- Use shared UI and layout patterns before creating new styles. Authenticated pages must match `docs/13_frontend_design_standards.md`; avoid ad hoc layouts, blue primary actions, and hardcoded UI strings.
- Prefer lucide-react icons over inline SVGs when an equivalent icon exists. Keep manual SVGs only when no suitable shared/library icon exists.
- Keep comments useful and sparse. Existing comments often reference requirement IDs; preserve or add REQ references when implementing requirement-driven behavior.
- Do not commit secrets, local logs, build outputs, generated `bin/obj`, `.next`, or environment files.

### Development Workflow Rules

- Work from the real repository root: `B:\Projects\IAB Connect\iab-connect`. The parent folder is not the Git repository.
- Use feature branches for changes and keep `main` stable. PRs require green build/tests, no secrets, a short description, and a requirement ID when applicable.
- For backend development, run commands from `backend`: `dotnet test` for all backend tests and EF commands with `--project src/IabConnect.Infrastructure --startup-project src/IabConnect.Api`.
- For frontend development, run commands from `frontend`: `npm run dev`, `npm run typecheck`, `npm run lint`, `npm run format:check`, `npm test`, and Playwright commands as needed.
- Local infrastructure starts with Docker Compose from the repository root using `infra/docker-compose.yml`. PostgreSQL, Keycloak, RustFS, MailHog, and related services are infrastructure, not application code.
- Backend and frontend must run in separate terminals: backend from `backend/src/IabConnect.Api` with `dotnet run`, frontend from `frontend` with `npm run dev`.
- EF Core migrations must have descriptive names and be reviewed before production. Apply migrations in staging before production.
- Deployment expects controlled backend migrations, Docker/container infrastructure, separate secrets per environment, monitoring/logging, and backup/restore validation.
- Use Conventional Commits style from the README when making commits: `<type>(<scope>): <description>`.
- BMAD artifacts belong under `_bmad-output`; do not mix planning/implementation artifacts into source folders unless they are intended project documentation.

### Story Authoring & Dev-Story Execution Rules (from Epic-11 retro A28-A30, 2026-05-16; Epic-12 retro A31-A33, 2026-05-16; Epic-20 retro A34-A37, 2026-06-01; Epic-13 retro A38-A42, 2026-06-01; Epic-15 retro A43-A45, 2026-06-02; Epic-16 retro A46-A48, 2026-06-02)

- **Spike-First for "low-risk mechanical" cleanup specs.** When a story AC describes a refactor as "low-risk mechanical", "trivial cleanup", or similar low-friction language (such as E11-S2 AC-2's appsettings.json base cleanup that hit a Hangfire eager-init blocker), the dev-agent MUST execute a `Task 0: Spike` that reads all consumers and identifies init-timing constraints BEFORE the cleanup subtasks begin. Spike output is one line: either "Confirmed low-risk → proceed" OR "Blocker found: <description> → escalate scope".
- **AC-Subitem Completion Check at Story Close.** When an AC enumerates "N items" (for example "tests assert 5 hardenings: Swagger 404, Hangfire 404, HSTS, HTTPS redirect, strict CORS"), the Quality-Gates closing Task MUST explicitly list each sub-item's status (covered / deferred / N/A). Aggregate claims like "all 5 verified" without per-item evidence are insufficient — they hide partial completion until epic-boundary review.
- **Three-State Task Checkbox for Manual-Verify ACs.** Use the explicit convention: `[x]` = dev-agent verified · `[!]` = needs human verify (manual smoke, browser interaction, infrastructure stand-up, dev-API not interactively launchable) · `[ ]` = pending. Manual-verification tasks bound by dev-agent non-interactivity get `[!]` (not `[x]`) so review-tracking sees the human-verification queue. The convention also lives in `docs/07_dos_donts.md`.
- **Cross-Story Orthogonal-AC Inventory at create-story time.** Stories in the same wave often share invariants that no single AC text enumerates: image-surface (no dev secrets in published OCI image), build-arg propagation (every Dockerfile `ARG` has a producer story + consumer story), env-var consistency (server-side env vs build-arg-baked env for the same logical value carry byte-equal strings), health-gate completeness (every long-running container has HEALTHCHECK + every dependent service uses `service_healthy`). `bmad-create-story` MUST add a Quality-Gates row per orthogonal dimension the story touches even when the AC text doesn't enumerate it. Pattern surfaced in E12-S1 (P2: `appsettings.Development.json` shipped in image despite AC-7's intent — AC-7 only covered base `appsettings.json` strings-grep, not the runtime-image filesystem inventory).
- **Decision-Resolution with Manual-Verify Hand-off.** When a code-review Decision-Needed finding can only be falsified via browser/manual/full-stack interaction, the Decision options presented via `AskUserQuestion` MUST include a "(X) verify-first via [specific manual step]" branch. If the user picks verify-first, the resolution is `resolved-pending-verify` (not `resolved`); the Story Tasks/Subtasks queues an `[!]` marker that references the Decision-ID; the code-review Findings section tracks these in a "Decisions Pending Verify" sub-bucket separate from Resolved and from regular Defers, so the Beta-deploy gate sees them. Pattern surfaced in E12-S4 D3 (NEXTAUTH issuer divergence — unverifiable without browser smoke, resolved via verify-first queued behind Task 8 `[!]`).
- **Bulk spec-refresh at epic start when implementing multiple stories in one session (A34).** When the user requests N≥3 stories in one session, `bmad-create-story` refresh runs as a single front-loaded batch BEFORE any `bmad-dev-story` starts. All spec-vs-reality deltas (broken line anchors, dead third-party action references, contradictory prose in surrounding files) surface as a batch — the user signs off on the deltas, then dev-story for each story runs without late-discovered drift. Pattern surfaced in Epic-20 retrospective (e20-s1's `dcoapp/action` 404 + e20-s2's README license-section contradiction would have been caught upfront, not as mid-flight scope additions).
- **Frontend Vitest + Testing Library: `afterEach(cleanup)` is mandatory for tests that call `render()` — and ONLY for those tests (A35, refined by A46 2026-06-02).** Vitest in this project does NOT auto-cleanup between tests. Every `render(<Component />)` from `@testing-library/react` accumulates DOM in the shared `document.body`. Tests using `screen.getByText(...)` will fail with `getMultipleElementsFoundError` from the second test onwards. **Apply when**: a test file imports `render` from `@testing-library/react` and calls it inside an `it(...)` block — include `import { cleanup } from "@testing-library/react"; afterEach(cleanup);` + the `// @vitest-environment jsdom` directive at file top. **Do NOT apply when**: the test does pure-Node work (file IO + regex assertions, mocked-module assertions, contract assertions over string content). Examples in repo: `BetaBanner.test.tsx` + `LicenseFooter.test.tsx` correctly use cleanup; `dockerfile-public-vars.test.ts` (reads `frontend/Dockerfile` via `node:fs.readFileSync` + regex matches) + `auth.logout.test.ts` (mocks `next-auth/react#signOut` + asserts callbackUrl) correctly do NOT. Pattern surfaced in E20-S4 LicenseFooter.test.tsx initial-failure-then-fix cycle; A46 refinement surfaced in Epic-16 retro 2026-06-02 when over-broad A35 application would have forced unnecessary jsdom-env directives on pure-Node tests.
- **ASP.NET Core integration tests reading env-var-mapped `IConfiguration` MUST override in `TestWebApplicationFactory` (A36).** ASP.NET Core's default `IConfiguration` chain reads unprefixed env vars (e.g., `BUILD_SHA`, `BUILD_DATE`, `PATH`, anything else). A test asserting `configuration["FOO"]` returns null is environment-dependent — it passes locally on a clean shell and flakes the moment CI exports the variable. Fix: in `TestWebApplicationFactory.ConfigureAppConfiguration`, explicitly bind empty values for any env-var-mapped key the test reads — `AddInMemoryCollection` runs LATER than the env-var provider so it overrides. Example: `["BUILD_SHA"] = string.Empty, ["BUILD_DATE"] = string.Empty`. Pattern surfaced in E20-boundary code review P4 (About_UnknownWhenEnvVarsMissing would have flaked the moment the E20-S5 GHCR workflow exported BUILD_SHA on the runner).
- **License-related changes must grep README for contradictory prose (A37).** When a story changes the project's license (LICENSE file, badge, SPDX identifier, or any license-named string), the dev-agent must grep README + top-level docs for license-related text and fold any contradicting prose into the same story. Grep patterns: `proprietary|All Rights Reserved|Private` (red flags); `License|Licensed|MIT|GPL|Apache` (review for accuracy). Pattern surfaced in Epic-20 retrospective (E20-S1 shipped the AGPL LICENSE file while README still claimed "**proprietary software** ... All Rights Reserved" — direct contradiction in primary marketing page; E20-S2 had to fix it as scope addition).
- **Doc-bundle pattern: when N stories all ship sections of one logical operator-facing document, write the complete doc once in the first story (A38).** When the user requests multiple stories whose only repo-tracked artifacts are sections of one logical document (deployment checklist, security runbook, migration guide, etc.), the dev-agent writes the **complete** document in the first story (with stub sections for the other stories' content if needed), and each subsequent story closes by pointing its Quality-Gates evidence column at the relevant section anchors — NOT by appending. **Reason:** appending in N stories produces N section-numbering inconsistency risks, N lint-style drift opportunities, N Markdown-link-anchor checks; writing once and pointing back is strictly cleaner. **Anti-pattern:** don't apply this when the doc is genuinely topic-bounded per story (a per-feature design doc, a per-component API reference). The pattern works when the doc is operator-facing and the stories carve up its sections. Pattern surfaced in Epic-13 retrospective (4 stories' deliverable was one 775-line `docs/14_beta_railway_setup.md`; the doc-bundle approach produced zero section-numbering inconsistencies at code-review time).
- **When documenting Keycloak env vars, grep the realm JSON for ALL `${...}` placeholders + `"bearerOnly": true` clients (A39).** Required Task 0 sub-step for any Keycloak-env-var-touching story: (1) `grep -oP '\$\{[A-Z_]+\}' infra/keycloak/realms-beta/iabconnect-realm.json | sort -u` to enumerate every realm-import substitution placeholder — every one needs a producer-side env-var row in the doc; (2) `grep -B 5 '"bearerOnly": true' infra/keycloak/realms-beta/iabconnect-realm.json` to identify bearer-only clients — these have no secret and must NOT carry a `*_ClientSecret` consumer-side env-var row. **Reason:** Epic-13-S2 documented three `_CLIENT_SECRET` placeholders but missed `${IABCONNECT_BETA_HOST}` + `${FRONTEND_PUBLIC_URL}` (would fail first browser smoke with `Invalid parameter: redirect_uri`); also prescribed `Keycloak__ClientSecret` for the bearer-only `iabconnect-api` client (a Sealed secret nothing reads). Both caught at epic-boundary review.
- **Verify shell-command syntax for tools the dev-agent doesn't exercise in-session, OR mark `[!] verify before executing` (A40).** When the doc prescribes a recovery / operations command for a tool the dev-agent has not directly exercised in the session (Keycloak CLI, Railway CLI, Postgres CLI in non-trivial modes, Docker swarm operations, cloud-provider CLIs), the command must be either (a) verified against the tool's current docs with the doc anchor cited, OR (b) explicitly marked `[!] verify against <tool> docs before executing`. **Reason:** Epic-13-S4 §11.2 originally prescribed `kc.sh bootstrap-admin --realm master --username recovery --password '<literal>'` — three things wrong: subcommand in Keycloak 26 is `bootstrap-admin user` (not bare `bootstrap-admin`), no `--realm` flag exists, and `railway run` does not shell into the container. Invented-but-plausible recovery commands are worse than no recovery procedure — they consume operator time during a real incident before they're discovered to not exist.
- **A32 Decision-Resolution: explicit autonomous-mode escape clause (A41, refines A32).** A32 step (d) (surface Decision-Needed via `AskUserQuestion` at Task 0) may be skipped when ALL THREE preconditions hold: (1) the user has **pre-declared autonomous-mode** in the session via explicit language ("no stopping", "implement them full", "be autonomous", "don't ask, just do" or equivalent), (2) the story file's Decision-Needed block contains a **recommended option** with documented rationale, AND (3) the dev-agent records steps (a) (the option chosen), (b) (the rationale, including a verbatim quote of the user's autonomous-mode directive), and (c) (the consequence chain — what AC sub-items flip from N/A to covered) in the Dev Agent Record → Debug Log References before proceeding. If ANY precondition is missing, A32 step (d) AskUserQuestion is still mandatory (no implicit escape). The escape clause does NOT override project memory `feedback_decisions_via_ask_tool` for choice prompts NOT tied to a Decision-Needed (e.g., post-implementation "apply patches" / "next step" choices remain AskUserQuestion). Pattern surfaced in Epic-13-S4 DEC-1 auto-resolution under "no stopping just straight forwards" directive + story-recommendation Option A; Acceptance Auditor flagged the (d) skip as soft-procedural; A41 path B chosen 2026-06-01.
- **Reread-as-a-stranger pass for documentation deliverables (A42).** AC-Subitem Completion Check (A29) verifies each AC's literal completion but does not catch cross-section contradictions, pre-filled placeholders that should be blank, stale anchor citations, or scope-leak from internal-process notes into operator-facing content. For any story whose primary deliverable is documentation, the dev-agent runs a fresh-eyes re-read pass before flipping to `review`, looking specifically for: (1) cross-section contradictions (e.g., README "immutable images" + "moving tag" in adjacent sentences); (2) pre-filled status checkboxes that turn a reusable checklist into a one-deploy snapshot; (3) stale file:line anchors carried from older story drafts; (4) imprecise claims that contradict the actual artifact (e.g., "5-line route" when the file is 8 lines); (5) sprint-tracking commentary leaked into operator-facing sections; (6) **documented-binary-surface reachability (per A45)** — if a doc snippet invokes a non-trivial binary, the binary must be reachable from the documented invocation site (verify against the Dockerfile / installation script / image-build pipeline; if the binary is operator-provided, the doc says so explicitly). This is what the Blind Hunter does at epic boundary — pulling it forward to story close means epic-boundary review catches structural issues without burning cycles on surface-level slips. Pattern surfaced in Epic-13 retro: Blind Hunter found 17 doc findings, ~10 of which were this category.
- **(a)/(b)/(c) Debug Log template for A32/A41 Decision resolutions (A43).** When a story has a `Decision-Needed` (DEC-N) block and the user has pre-declared autonomous mode (A41), the dev-agent records the resolution in Dev Agent Record → Debug Log References using **exactly** this three-bullet shape: (a) **Option chosen** = one option-letter + 1-line summary; (b) **Rationale** = at least three short sub-claims (story recommendation + user autonomous-mode verbatim quote + downstream architectural justification); (c) **Consequence chain** = which ACs / sub-items flip in scope or out of scope. **Reason:** E15 exercised A41 across two DEC-1s (E15-S3 + E15-S4); both Dev Agent Records independently converged on this three-bullet shape but with slightly different ordering. Pinning the template makes future audits + future agents reading these as examples consistent. Also: when a Decision is later challenged in retro, the (a)/(b)/(c) record is a structured deposition.
- **Hangfire recurring-job-id global-uniqueness test (A44).** When a new Hangfire recurring job is registered in `Api/DependencyInjection.cs`, the registering story MUST add (or extend) a test that asserts the full set of project-wide job-ids is distinct. **Reason:** Hangfire's `AddOrUpdate` is keyed by id; two registrations sharing an id silently overwrite each other in the recurringjob storage table. Without this guard, a copy-paste regression (the most likely failure mode when adding a 7th job by templating from one of the existing 6) is detected only at the first 03:00 UTC deploy. Anchor: `RegisterDailyBackupJobTests.RecurringJobIds_AreGloballyUnique_AcrossAllRegisteredJobs` is the canonical test; future stories adding a new job-id constant extend the `ids` array + bump the `.HaveCount(N)` assertion. Pattern surfaced in Epic-15 boundary review patch P8.
- **A42 sixth category: documented-binary-surface reachability (A45, refines A42).** A42's existing five categories (cross-section contradictions / pre-filled status / stale anchors / imprecise claims / sprint-tracking leakage) cover doc-internal consistency but not doc-vs-implementation-image reachability. When a documentation deliverable includes a shell command that invokes a non-trivial binary (CLI tool, container utility, language interpreter), the A42 reread-as-a-stranger pass MUST additionally confirm: (1) the binary is reachable from the documented invocation site — verify by reading the relevant Dockerfile / installation script / image-build pipeline; (2) the binary version matches the spec contract — if the doc says "uses postgresql-client-17 features", verify the Dockerfile installs v17 specifically; (3) if the binary is operator-provided (must install locally), the doc says so explicitly — not silently assumed. **Reason:** Epic-15 boundary review P2 caught Section 15.4 + 15.5 Step 1 prescribing `aws s3` commands against a backend container that does not install the AWS CLI. The in-story A42 pass focused on cross-section consistency + redaction + anchor freshness, but did not cross-check the documented commands against the actual binary surface of the runtime image. A45 extends A42 to cover this surface as a mandatory sixth check.
- **A41 autonomous-mode escape extends to the live-walkthrough `[!]` queue, not just DEC-Needed (A47, extends A41).** A41 originally framed the autonomous-mode escape for `Decision-Needed` AskUserQuestion skips at Task 0. When a story carries a substantial `[!]` live-walkthrough queue (live Beta deploy + browser session + Railway dashboard + Keycloak Admin Console steps the dev-agent cannot orchestrate from its sandbox), the same escape MAY be applied uniformly to the entire queue — flipping every live-walkthrough AC from `covered` to `deferred-pending-beta-green` in the Quality-Gates Closing table — when ALL THREE preconditions hold: (1) the user has **pre-declared the deferral** in this session via explicit language ("i wont do the hard prerequisits yet", "i will do everthing at the end once im finished with all epic", "do everything until I tell you to stop" or equivalent); (2) **each story file's design pre-acknowledges the live-Beta gate** as a Task 0 `[!]` prerequisite + paste-blanks in the doc-bundle deliverable; (3) the dev-agent records a single (a)/(b)/(c) block PER STORY in Dev Agent Record → Debug Log References (per A43) that quotes the user's deferral verbatim AND enumerates the row-flips as the consequence chain. The Quality-Gates table MUST distinguish `covered` vs. `deferred-pending-beta-green` per row — no aggregate claims. The unified human-verify queue (a "Q1..QN" table in each story's Completion Notes) is the operator-facing artifact that surfaces the deferred items at execution time. **Phrase coined and pattern surfaced** in Epic-16 retro 2026-06-02 (3 verification-only stories with ~30 live-Beta `[!]` items deferred to a unified end-of-Wave-8-or-9 session). **Does NOT override** project memory `feedback_decisions_via_ask_tool` for choice prompts NOT tied to a deferred `[!]` queue (e.g., post-implementation "apply patches" / "next step" prompts remain AskUserQuestion).
- **Refactor `Program.cs` Serilog setup to be `WebApplicationFactory`-idempotent (A49).** The current `Log.Logger = new LoggerConfiguration()...CreateBootstrapLogger()` at Program.cs:8-11 + the `try { ... } finally { Log.CloseAndFlush(); }` envelope means a second `WebApplicationFactory<Program>` instance in the same process trips "logger is already frozen". This forced ~3 E14 stories (E14-S2 HSTS + HTTPS-redirect, E14-S3 Beta-env-derived factory, E14-S4 named-policy + 429 transition tests) to pivot from runtime tests to code-audit regex tests — leaving ~6 ACs partially-met-via-loophole. **Fix**: wrap Serilog setup in a re-entrancy-safe pattern (`Log.Logger ??= ...` or a static lazy initializer that skips on re-entry, or a `ProgramFixture` test pattern that resets `Log.Logger`). When fixed, retroactively add the deferred runtime tests for HSTS max-age value, HTTPS 307 + Location, 429 transition + Retry-After + CorrelationId propagation. - **Build-time env vars in security-relevant strings MUST be parsed/normalized, not raw-concatenated (A50).** When a build-time env var (NEXT_PUBLIC_* / build-arg / GHA repo variable) is interpolated into a security-relevant string — CSP directive, CORS-allowed-origin list, redirect-URI allowlist, Keycloak realm import `${VAR}` substitution, or any other policy-shaped string — the value MUST be parsed via `new URL()` + emit only `.origin` (or equivalent normalization for the format) and MUST reject values containing format-meaningful separators (semicolons + whitespace + quotes for CSP; commas for CORS; etc.) before interpolation. **Reason**: a compromised CI variable, a fork operator mis-typing, OR a copy-paste accident can otherwise inject new directives. Pattern surfaced in E30-S5 (AC-2/AC-6 spec-correction; the suppliers guard branch is correctly left inline).
- **A byte-identical mass-swap fanned out to N parallel subagents REQUIRES an orchestrator post-hoc completeness + consistency grep sweep (A110, extends A101).** The per-agent characterization nets do NOT catch a missed swap (the net doesn't assert the frame, A108) or a half-done one. After the fan-out, the orchestrator MUST grep for (a) the swap's structural signature still present unswapped (a PLAIN canonical frame the agents missed), and (b) the divergence signature (e.g. a file carrying BOTH the new primitive AND the old inline frame = a half-adoption). Pattern surfaced in E30-S5: 8 parallel subagents; the orchestrator's two greps caught 5 finance half-adoptions (reverted) + confirmed no plain canonical sole-child frame was missed — neither was visible to any green net.
- **A manual-fetch → TanStack migration must DECIDE the auto-refetch triggers (`refetchOnWindowFocus`/reconnect/mount), not only `retry`/`enabled` (A111, extends A79/A99).** A one-shot `useEffect`+`fetch` god-page never refetched on window-focus; the naive `useQuery` migration inherits TanStack's `refetchOnWindowFocus: true` and silently re-hits the endpoint on every tab-return past `staleTime`. A79's "enumerate the semantic deltas and decide which to preserve" MUST include the auto-refetch triggers, not just retry/loading/error stickiness. Set `refetchOnWindowFocus: false` (and reconnect if the god-page didn't reconnect-refetch) to preserve the single-fetch invariant. Pattern surfaced in E30-S4 boundary review P1 (the dashboard query set `enabled`+`retry:false` but missed the focus-refetch).
- **prettier-CLEAN files INVERT the A72 `--write` ban (A112, refines A72/A58/A81).** A72 bans `prettier --write` on a MODIFIED file because the file is assumed PRE-drifted (the repo carries ~2543 drifted files) and `--write` would balloon a logical edit into a whole-file reformat. But files written cleanly by a prior migration (the E22-E29 slice files) are prettier-CLEAN — and after a structural edit that removes a nesting level (e.g. a `<main><div>`→`<PageShell>` frame swap), the dedented children are now mis-indented, so `prettier --write` is REQUIRED (it only re-indents what the swap disturbed; there is no pre-existing drift to balloon). **Rule:** determine each work-list file's drift status FIRST (`prettier --check`); on a CLEAN file `--write` after a structural edit is correct, on a PRE-drifted file the A72 ban + hand-match still applies. Pattern surfaced in E30-S5: all 74 retrofit-target files were prettier-clean (the OPPOSITE of the story's "73 pre-drifted / A72 is the headline risk" framing).

- **Retiring a WRAPPED legacy module = the INVERSE of A94/A106: LIFT the exact bytes into the owning slice + repoint importers + retarget the `vi.mock` seams; the pre-existing suite is the oracle (A87 inverted) (A113).** Every transport the domain epics (E22-E30) kept in `@/lib` survived via the A94 WRAP / A106 live-binding re-export precisely so the old `vi.mock("@/lib/...")` seams kept intercepting. To RETIRE it (E31), do the reverse: copy the lib module's bytes verbatim into `features/<domain>/api/<resource>.ts` (a NEW prettier-clean file — `--write` it), repoint every importer's specifier, and **retarget every `vi.mock`/`importActual`/type-import** from the `@/lib/...` path to the new path. Behaviour is byte-identical BY CONSTRUCTION (bytes unchanged, only location + specifiers); there is NO new char-net — the accumulated suite IS the oracle, green-after-repoint is the proof. The mock-retarget across ~80 test files is the BULK of the work, not the transport relocation. A clean per-cluster verification (`grep` the legacy path → 0 live importers + run the cluster's tests) localizes any missed seam. Pattern: E31-S1 retired 17 legacy modules across 10 clusters this way; the suite held at 2013 green throughout. Parallel subagents can own independent clusters (A87/A101) IFF the orchestrator pre-creates the shared `@/types/*` homes first (else they race on the shared file).
- **A mechanism-CHANGING relocation that NO test exercises needs an EXECUTABLE byte-identity net authored at dev time (A114, refines A113/A88).** A113 relocation is byte-identical when the bytes don't change — but when the MECHANISM changes (E31: the class-based `@/lib/api-client` `ApiClient` was retired and its semantics INLINED into a private `fetch` `request` helper inside the `email-templates` slice), and EVERY consumer mocks the module (so no test runs the real fetch), the suite is NOT the oracle for that transport — its "byte-identity" rests on manual read + `tsc` alone. The net-integrity auditor can only assert it by eye. **Rule:** when a relocation rewrites the transport mechanism AND the real implementation has zero executing coverage, author a focused fetch-spy test pinning the exact URL/verb/headers/body/`{}`-for-204/error-throw-shape at dev time (don't wait for the boundary review to flag it as residual-risk). Pattern: E31 `emailTemplatesApi` `ApiClient`→`fetch` rewrite — Edge-Case + Net-Integrity both flagged the coverage gap; the boundary-review patch added a 9-test byte-identity net (suite 2013→2022).
- **Shared-symbol home decision tree when retiring a shared module (A115, consolidates DEC-2/3/4).** When a retired transport's symbols are consumed by ≥2 features, classify each symbol and route it: **(1) pure types/enums/leaf-helpers** (no fetch) → a shared `@/types/*` module (DEC-2; import-legal from any feature, NO `eslint-disable`) — this is the MAJORITY and routing it here keeps the cross-feature-import count near zero; **(2) the shared TRANSPORT fns** (the actual fetch) → ONE owner slice; the 1-2 sibling consumers cross-import with an explicit `// eslint-disable-next-line no-restricted-imports` + reason (DEC-3); **(3) a generic transport BASE** (e.g. an `apiGet/apiPost`+`ApiResult` helper) used by 2 unrelated owners → each-owner-its-own-copy is acceptable (DEC-4) to avoid coupling unrelated features for a generic util (the ~50-line duplication beats an `events→documents` dependency). Putting helpers+types in `@/types/*` (1) shrinks the `eslint-disable` surface to ONLY the transport-fn importers (2). Pattern: E31 documents 3-way (enums/DTOs/4 helpers→`@/types/documents`; 17 transport fns→`features/documents` + 2 sibling `eslint-disable`s; `ApiResult`→`@/types/api-result`; the http base copied into both documents+events).
- **A flat-config `no-restricted-imports` legacy-path ban MUST be spread into every zone rule + a catch-all that EXCLUDES the zones — a separate repo-wide rule CLOBBERS the zones (A116).** ESLint flat config resolves a given rule by LAST-MATCH-WINS (rule values replace, they don't deep-merge). So a new repo-wide `{ files: ["src/**"], rules: { "no-restricted-imports": <legacy-ban> } }` placed after the existing per-zone `no-restricted-imports` rules (components/ui, components/layout, lib, features) would OVERWRITE each zone's cross-zone ban for its files. **Rule:** define the ban as a shared `patterns` entry, SPREAD it into every existing zone rule's `patterns` array, AND add a catch-all config (`files: ["src/**/*.{ts,tsx}"]`, `ignores: [the 4 zone globs]`) for app/types/hooks/etc. so no file is matched by two `no-restricted-imports` configs. ALWAYS spot-verify the ban FIRES: a throwaway `import "@/lib/<banned>"` in BOTH a zone-covered file (e.g. `src/features/`) AND a catch-all file (e.g. `src/app/`) must error, then revert. Do NOT ban live infra (`@/lib/auth`/`@/lib/modules`/`@/lib/utils`). Pattern: E31-S2 `legacyHttpClientBan`.
- **Importer-count-ZERO (the build-safety gate) is distinct from string-reference-ZERO (a cosmetic sweep) — gate on the former, sweep the latter separately (A117).** After deleting a module, the BINDING "nothing references the deleted path" gate is: the import-statement grep (`from "X"`/`vi.mock("X")`/`import("X")`) returns zero AND `tsc --noEmit` + `next build` exit 0 (the production build is the definitive dangling-import backstop — `tsc`+vitest can pass while a dynamically-resolved import dangles). SEPARATELY, historical/provenance COMMENT references to the deleted path remain (e.g. "relocated off `@/lib/api/foo`", "the spec that `vi.mock(\"@/lib/...\")` keeps intercepting"); a literal "zero references" AC wants those swept too — do it comment-only (capture-based `sed`, re-verify `tsc` after, since no code uses the token once importers are zero). Don't conflate the two: a comment hit is never a build blocker, and the import-grep + build is what actually proves safety. Pattern: E31-S2 swept ~119 comment refs after the importer-grep already showed zero; the final scan then read clean, but the load-bearing gates were tsc + `next build`.

### Critical Don't-Miss Rules

- Do not treat UI role checks as security. Every protected backend operation needs policy/permission enforcement and sensitive failures should be audit logged.
- Do not add microservices, message brokers, or separate deployables for MVP work unless explicitly requested. The architecture decision is modular monolith first.
- Do not put business logic in EF entities, API endpoints, or frontend components when it belongs in Domain/Application services, MediatR handlers, validators, or repositories.
- Do not bypass existing finance/accounting compliance patterns: finance deletes are often soft deletes, sent/overdue invoices use cancellation/reversal behavior, and double-entry behavior is controlled by finance profile/accounting mode.
- Do not manipulate private backing-field aggregate collections and assume EF will persist new child entities. For known problematic aggregate child persistence, add child entities directly to the relevant `DbSet` as documented in `docs/07_dos_donts.md`.
- Do not change requirement source content in the CSV unless explicitly asked. Update requirement status separately and preserve REQ IDs.
- Do not introduce hardcoded UI text, German UI literals, or enum values that differ from backend contracts. UI text goes through next-intl and enum values must match PascalCase API values.
- Do not use blue as the primary action/link color in authenticated UI. Use orange-600/orange-700 and the standard page layout.
- Do not add list/table pages without search/filter controls.
- Do not refresh frontend data by chaining duplicate fetches inside mutation handlers. Use a refresh trigger state and effect/cancellation pattern.
- Do not use EF InMemory as proof that PostgreSQL repository behavior is correct. Use Testcontainers for relational behavior, concurrency, query filters, constraints, and migrations-sensitive logic.
- Do not weaken privacy/retention/audit behavior for convenience. Member, finance, document, backup, search, and export features can expose sensitive data and must preserve authorization, audit, and retention rules.
