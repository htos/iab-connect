## 2. What worked

1. **Hybrid workflow validated itself.** Skipping per-story `bmad-code-review` and bundling CR+ER at the epic boundary surfaced 11 patches, of which 5 were critical (AC violation, test regression, path-traversal symmetry gap, user-lockout bug, null-Id filter gap). All would have shipped without the boundary review.
2. **Test-driven recovery on the GUID-validation regression (S4.3).** The 5 broken tests were not just repaired with valid GUIDs — they were extended with `[Theory]` cases (5 inline rows each, including empty/whitespace/non-GUID/prior-literal/path-traversal-attempt) that lock the validation contract in. Pattern was then applied to S2.2 (`ResetUserMfaAsync` + `SendPasswordResetEmailAsync`). Net: **+30 tests** across the patch pass.
3. **Sprint Change Proposal preserved context across context-window boundaries.** `sprint-change-proposal-2026-05-13.md` carried the full state (file / where / change / rationale, per patch) through three separate invocations: `bmad-correct-course` → first `bmad-dev-story` (E1.S4) → second `bmad-dev-story` (E1.S3 + E1.S2). Cold-start readability was high.
4. **Architecture decisions held.** Keycloak remained the identity authority (no local MFA secrets, no local session store). Backend authorization stayed the security boundary. Modular monolith / Clean Architecture borders intact. Sensitive actions audit-logged via `SecurityAuditLogger`.
5. **Real Keycloak validation evidence.** E1.S1 captured admin-REST-API structural assertions on the live realm AND a user-executed browser flow (`admin/kassier/member` logins, TOTP enrollment, Keycloak event evidence). Not just JSON-shape tests.

## 3. What didn't work

1. **Per-story code review was skipped → risk concentrated at the boundary.** Applying 11 patches against 3 stories at once is more cognitive load than 2–3 patches per story at story-close. The S4.3 test regression sat in the working tree from P1 onwards and would have failed CI silently until the boundary review.
2. **Uncommitted P1–P6 fixes in the working tree before patch pass.** `IdentityEndpoints.cs` and `KeycloakAdminService.cs` carried partial uncommitted P-fixes. Without commit discipline at story-close, "current state" became ambiguous.
3. **P1 GUID validation was asymmetric.** Initial P1 fix only guarded `GetUserSessionsAsync` and `RevokeSessionAsync`. `ResetUserMfaAsync` / `SendPasswordResetEmailAsync` (and other userId-interpolating methods) stayed exposed until S2.2.
4. **Logger category was wrong twice before correct.** Original: `ILogger<KeycloakAdminService>` injected into `IdentityEndpoints`. P5 changed to `ILogger<Program>` — also non-specific. S4.1 introduced the `IdentityEndpointsLog` marker class because the static class can't be a `TCategoryName`. Pattern not recognized at original-implementation time.
5. **`SessionMapper` was initially `internal`.** S3.1 (internal-Keycloak-client filter) required promotion to `public static` so it could be exercised from the API-tests assembly. Mappers on the HTTP path are inherently test-public — designing for that upfront avoids the post-hoc promotion.

## 4. Cross-story patterns

- **Audit-log verb misuse:** `LogAccessDenied` was used for infrastructure errors (S2.1 fix), missing on 404 catch (S3.2 fix), and missing on idempotent admin-revoke 404 (S4.4 fix). Lesson: `LogAccessDenied` = permission refusal; `LogAccessGranted` = state-changing action succeeded; infrastructure failures belong in the standard `ILogger`.
- **Keycloak edge defensiveness:** empty/null session IDs (P6 + S3.4), 400 from `execute-actions-email` (S2.3), non-GUID userId strings (P1 + S2.2). Pattern: Keycloak Admin API is not strict — defensive code is required at both inputs and outputs.
- **Self-service vs. admin symmetry:** P6 (null-Id filter) only on self-service, S3.4 fixed admin path; P3 (idempotent audit on 404) only on self-service, S4.4 fixed admin path; P1 (GUID guard) partial. Lesson: mirror endpoints need mirror logic — or a documented reason for asymmetry.

## 6. Hybrid-workflow assessment

**Recommendation: keep the hybrid workflow with an escalation trigger.**

- **Keep (skip per-story `bmad-code-review`)** for standard stories (backend endpoint + tests + UI touch).
- **Escalate to per-story** when a story
  - touches sensitive workflows (finance, privacy, audit-critical), OR
  - changes more than ~300 LoC, OR
  - extends cross-cutting code (e.g., `KeycloakAdminService`-class services).

For Epic 2: E2.S3 (Safe Member Merge) hits all three → per-story review (action item A5). E2.S1, S2, S4 are standard → bundle at the epic boundary.
