## What went well (apply forward)

- **The area-sub-slice fan-out (A101) is the right shape for a broad area.** Splitting 15 pages into 5 independent slices let 5 parallel agents extract simultaneously with no file contention, each gated on its OWN S1 sub-net (the regression oracle) + its slice tests, and the orchestrator ran the single full `tsc`/`vitest`/`eslint` pass centrally. The S1 net (written first, green before any refactor) was the behaviour-preservation proof for all 5 migrations at once.
- **A94 wrap-vs-build decided per sub-area, correctly** (3 WRAP / 2 BUILD). The WRAPs needed ZERO S1 test edits; the BUILDs reused the already-mocked `useApiClient` seam — so across 5 slices the S1 net needed only ONE licensed assertion change (S4 DEC-4) + a harness wrapper (S5).
- **The prior epics' lessons held.** A95/A96/A98 forms (no `.trim()`, per-field errors, mode-divergent props threaded), A99 sentinel-by-transport (`retry:false` where the wrapped fn carries no status), A90/A97 guards, A86 affordance colours — all applied up front by the slice agents; the boundary review found far fewer behaviour-preservation gaps than E25 (4 vs 9 patches), and the only MED was a new class (A100), not a recurrence.
- **The harness translator-stability fix (A64 refinement).** The admin pages put `t` in `useCallback`/effect dep chains, so the S1 nets had to mock `next-intl` with a MODULE-STABLE translator (not a fresh arrow per render) to avoid infinite render loops / non-deterministic refetch counts — same for a stable `useApiClient` object identity (A78). Confirmed across the system + integrations nets.

## What the boundary review caught (the durable lessons)

The net was green + tsc + eslint + prettier all passed, yet the adversarial review found **4 behaviour-preservation / consistency gaps** the net didn't cover. Two generalize into new action items; one reinforces A95.

### A100 — optimistic-overlay reset must key on the query FILTER, not `data` identity; and a comment claiming an invariant must be implemented (MED; from E27-S2 boundary P1; refines A79/A94)

The admin-users god-page mutated its local list in place on delete/toggle WITHOUT a refetch, and re-seeded the whole list from the server on every search/page fetch. The slice migrated this to a TanStack query whose displayed rows are DERIVED during render from the query data + two local overlays (`deletedIds`, `overrides`) — the A79 way to avoid a setState-in-effect mirror. But the overlays were NEVER reset, despite a comment claiming they "reset on a new fetch" — so a stale optimistic disable could mask fresher server data after a navigation, and the comment lied. **Rule:** reset the overlays when the active FILTER (page/search) changes — keyed on the FILTER, NOT on `data` identity, because the mutation hooks invalidate the list and a SAME-KEY refetch arrives with new `data`; a data-identity reset would clear the just-applied optimistic delete and the S1 delete spec's deleted row would flash back. Implement via React's adjust-state-during-render pattern (`if (seenKey !== filterKey) { setSeenKey(filterKey); resetOverlays() }`), never a setState-in-effect. Companion discipline: a comment asserting such an invariant MUST be backed by code — this passed the green net + tsc + eslint and was caught only when the adversarial auditor read the code against the comment.

### A101 — when an AREA is migrated as N parallel mutually-independent sub-slices, add a CROSS-SLICE CONSISTENCY pass (the area-sub-slice pattern; from E27 P3+P4; extends A91)

Fanning a broad area out to parallel agents is correct for throughput (no shared production files → no conflicts; the generic `src/features/**` boundary needs no per-feature entry), BUT each agent resolves the area's SHARED sub-decisions in isolation, so they diverge. E27 shipped one slice's `webhook-deliveries` page WITHOUT the non-admin `return null` terminal every sibling had (P3), and one slice kept a `vorstand` role badge blue while the sibling remapped the SAME role blue→amber per the "no blue in authenticated UI" rule (P4). **Rule:** when fanning an area out to parallel sub-slices, (a) pin the shared idioms — guard/`return null` shape, destructive-affordance + badge-colour decisions, DEC resolutions, naming — in ONE brief the agents follow, and (b) add a cross-slice-consistency layer to the boundary review ("does every sub-slice resolve each shared sub-decision the same way?"). Per-slice correctness is necessary but not sufficient.

### A95 reinforced — rendering the out-of-set `<option>` is necessary but NOT sufficient; the Zod field must be `z.string()`, never `z.enum(subset)` (from E27-S4 boundary P2)

A95 (RHF+Zod migration of a manual `<select>` must round-trip an out-of-set stored value) RECURRED: the retention form rendered the extra `<option>` for an out-of-set `action` but kept `action: z.enum([…3 offered])`, which would still REJECT the out-of-set value on submit — so A95 rule (1) "widen the Zod field to the FULL union / `z.string()`" is the load-bearing half, not the option rendering. Latent only (the backend enum is exactly the 3 today), but a real A95 violation; patched to `z.string()`.
