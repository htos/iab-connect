## What was tricky (lessons)

- **A correctness override beat the story's recommendation (DEC-2, S3).** The spec recommended a one-way hash for the webhook secret, but the server must *read* the secret to sign every outbound delivery — a one-way hash makes S4 impossible. Chose AES-256-GCM reversible-at-rest, documented in the Debug Log. Lesson: when a downstream story will consume a stored secret, "store a hash" is wrong by construction — verify the consumer before picking storage.
- **Claim-before-send can poison the DbContext (boundary-review P1).** A unique-key collision left the failed delivery tracked in `Added` state; a subsequent insert on the same context would re-attempt it. Fixed by detach-on-collision in the repo. The E5 reference implementation persists Pending rows in a batch and didn't surface this because its single `AddAsync` covered all rows; the per-subscription fan-out loop exposed it. → **A74.**
- **SSRF fail-closed trades availability for safety.** A DNS-resolution failure blocks the target with no retry. Correct default, but the transient-vs-confirmed distinction is a real nuance for any outbound side-effect guard. → **A75.**
- **The `react-hooks/set-state-in-effect` rule.** New admin pages tripped it; the clean pattern is an inline async IIFE that setStates only *after* the `await` (not calling a setState-bearing `useCallback` from the effect). Reused across all 4 pages.

## New action items (promoted to project-context.md)

- **A74** — Claim-before-send in a per-item fan-out loop must DETACH a failed row on a unique-key collision before continuing on the shared DbContext; otherwise the doomed `Added` entity re-attempts on the next `SaveChanges` and poisons an unrelated item. A run-twice test won't catch it (separate scopes); test a collision-then-distinct-insert on the SAME context.
- **A75** — An outbound side-effect guard (SSRF, allow-list) that fails CLOSED must decide retry semantics deliberately: confirmed-block → no retry; transient determination failure (DNS) → consider rethrow-for-retry. Default to fail-closed for security, but document the availability trade-off.
