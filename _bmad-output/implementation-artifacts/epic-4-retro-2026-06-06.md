## 2. What Went Well

- **A56 existing-implementation spike paid off again (4th epic running).** The create-story refresh correctly identified that `Event.Cost`/`IsFree`, the public registration page, the admin roster, and `formatCurrency` already shipped — so S1/S3 were authored as *extend + regress-guard*, not greenfield. Zero accidental rewrites of the legacy cost surface.
- **Atomicity was designed correctly and proven by test.** The S2 coordinator wraps fiscal-check → atomic invoice number → `Invoice.Create` + `AddItemWithTax` → one `SaveChangesAsync` → commit in an explicit transaction; the raw invoice-number UPSERT enlists in that transaction. Rollback is proven by Testcontainers tests (fiscal-lock + currency-mismatch → neither row persists). The hardest AC (AC-3) shipped solid.
- **Finance compliance reused, never reimplemented.** Cancellation disposition (Draft→soft-delete / Sent→Cancel / Paid→intact) routes through the existing `Invoice` domain methods; no new RecipientType; `Amount (18,2)` ↔ InvoiceItem precision parity held. The "Critical Don't-Miss" finance rules were respected.
- **Test-harness debugging produced two durable frontend-test lessons** (A63, A64) — the kind of friction that only surfaces the first time a feature epic adds DI params to a shared endpoint group and renders a `t`-in-deps page under Vitest.
- **Honest QGT + deferral discipline.** S2 marked the un-exercised module-gate HTTP test `[~]` rather than ✅; S3's completion notes flagged the member-UI gap. The epic-boundary review then promoted these to tracked E4-FT items rather than letting them hide.

## 3. What Was Hard / What We Learned

- **A sibling story's DEC is not a delivered contract.** S2-DEC-6 said S3 would "derive payment status from the linked invoice," and S3's spec framed its backend work as a "focused email edit." In reality S2 never exposed payment fields on the registration DTO or any public fee endpoint — S3 had to add a public AllowAnonymous fee endpoint + a roster invoice-merge + DTO fields. The spec under-scoped S3's backend because it assumed S2 had shipped a surface it hadn't. → **A62.**
- **Multi-aggregate atomicity forced an architecture pivot mid-implementation.** DEC-3 recommended a MediatR command in Application, but the per-aggregate repositories (`EventRegistrationRepository`, `InvoiceRepository`) each call `SaveChangesAsync` internally, and Application cannot reference `ApplicationDbContext` — so a pure Application handler cannot deliver single-commit atomicity. The coordinator had to be an Infrastructure service. This was the right call and is documented, but it's a recurring shape worth flagging at create-story. → **A61.**
- **Adding DI params to a shared endpoint group breaks sibling metadata-test harnesses.** Both `EventCheckInEndpointTests` and `EventCheckInRosterEndpointTests` map `MapEventRegistrationEndpoints`; when `RegisterPublic`/`RegisterMember`/`GetRegistrations` gained service params, minimal-API threw "Failure to infer parameters" at map time across those (unrelated) harnesses. → **A63.**
- **Vitest + a `t`-in-deps page hangs with a naive next-intl mock.** The fees page test initially never rendered its dialog because the mocked `useTranslations` returned a fresh function per render and the page keeps `t` in its load-`useEffect` deps → infinite reload loop. → **A64.**
- **An AC spanning two UI surfaces (public + member) is a coverage trap.** S3 AC-1/AC-3 cover both the public registration page AND the member registration surface; only the public path was built, yet the QGT marked them ✅. The member fee/payment UI is genuinely missing (E4-FT-2), making multi-tier member-priced events unfulfillable. → **A65.**

## 5. Action Items (promoted to project-context.md as A61–A65)

| ID | Lesson |
|---|---|
| **A61** | Multi-aggregate atomic coordinators belong in Infrastructure (direct DbContext + explicit transaction), NOT Application MediatR, when the per-aggregate repos each call `SaveChangesAsync`. Flag at create-story when an AC claims "one SaveChangesAsync" across ≥2 aggregates. |
| **A62** | A sibling story's DEC is not a delivered contract — when story N relies on data "from story N-1," verify N-1 actually shipped that surface (DTO field / endpoint), or scope the surface into story N. |
| **A63** | When a shared endpoint group's handler gains a DI service param, every endpoint-metadata test harness that maps that group must register the new service, or minimal-API throws "Failure to infer parameters" at map time. |
| **A64** | Frontend page tests must mock `useTranslations` to return a STABLE function reference — a fresh `t` per render re-fires any `t`-in-deps `useEffect` forever (the page never settles). |
| **A65** | An AC that spans two UI surfaces (e.g. public + member registration) is a coverage trap — build both surfaces or split into two ACs so the QGT can honestly mark one done and one deferred. Don't ✅ a half-built multi-surface AC. |
