## 3. What didn't work

### 3.1 Four review rounds is a signal, not a triumph

Epic 2 closed in one boundary review round. Epic 3 took four. Critically, **Round 3 was not a polish pass** — it surfaced 4 *fresh* Critical-severity defects independent of the Round-2 surface:

- CSV formula injection (R3-C1) — attacker-controlled `SpecialRequirements` from anonymous public registrants.
- Two cross-event IDORs (R3-C2/C3) — `PUT volunteer-roles` and `GET shift-assignments` discarded `eventId`.
- One-way `HashCalendarSubscriptionTokens` migration (R3-C4) — empty `Down()`, no recovery path.

The Round-2 review (12 reviewer-agents across 3 layers) missed whole defect classes. **Even Round 4 — a third adversarial pass — still found a High in S2 (500-not-409) and two Highs in S3 (UTC guard, list-query IDOR).** One boundary review is demonstrably insufficient for this risk profile.

**And A10 — the developer-judgment mid-epic escalation from the Epic-2 retro — never fired.** E3.S3's Round-2 review alone returned 19 findings; the trigger ("if accumulated patch backlog feels like ≥1 day of fix work, pause and run a review") should have caught it. A judgment-only trigger doesn't fire when the developer is heads-down.

### 3.2 Cross-event authorization / IDOR was the dominant defect class

It took **all four rounds** to fully close in E3.S3:
C1 (R2) authorization bypass — `CancelVolunteerAssignment` had no ownership check → R3-C2/C3 cross-event IDOR → R4-P-S3-2 list-query IDOR (enumerate Hidden/InviteOnly events) → R4-P-S3-5 fail-open guard (`callerMemberId == null` + non-staff caller passes).
The Round-2 H-S3-2 patch sweep "fixed" sibling commands but never abstracted the *pattern* — so each round re-discovered the same root cause on a different endpoint.

### 3.3 AC-mandated test files were simply not written

The Acceptance Auditor flagged missing, spec-named test files in **Round 3 and again in Round 4**:
- E3.S3 — 4 command-handler test files (incl. the 8-row capacity/waitlist `[Theory]`) — R4-P-S3-3/4.
- E3.S4 — `VolunteerAssignmentReminderQueryTests` + `VolunteerShiftReminderJobRegistrationTests` — R4-P-S4-3.
- E3.S5 — `EventCalendarFeedEndpointTests` + `GetPublicCalendarFeedQueryHandlerTests` — R3-H-S5-7/8, re-surfaced R4-Defer-S5-5.

The spec named the files. Dev skipped them. The public calendar feed's AC-1 visibility filter — the core privacy guarantee — still has **zero automated coverage** as of epic close.

### 3.4 Every backend story's planning artifact was wrong

The 2026-05-12 story templates for S1, S2, S3, S5 all listed net-new artifacts (`GetEventCheckInRosterQuery`, `CheckInRegistrationCommand`, the 3 volunteer entities, the ICS generator) as "existing code to inspect." Each story wasted dev time grepping to disprove it.

### 3.5 The Serilog static-state collision deferred real auth tests epic-wide

A second `WebApplicationFactory<Program>` re-runs `Program.Main` → "logger is already frozen" on static `Log.Logger`. S1, S2, S5 all hit it; all three replaced *runtime* API auth tests with metadata-only theory stand-ins. The genuine auth-behaviour tests are deferred.

### 3.6 A "pragmatic accept" didn't survive re-review

D-S4-1 accepted the shift form as-implemented (plain `useState`, inline `<section>`, no react-hook-form/zod/Radix) in Round 2. Round 4 overrode the accept and patched it anyway (R4-P-S4-1/2). A round was spent shipping something that was then un-shipped.

### 3.7 The deferred backlog more than tripled

26 → ~96 items. ~68 originated in Epic 3 alone. The bulk is low-severity / spec-text drift — but a real security-adjacent core sits inside it (see §8).

## 6. Workflow re-assessment

**Decision (Harry, 2026-05-14): the four-round count is a *dev-process* signal, not a review-process one. Fix the dev side; do not add a mandatory mid-epic review.**

Rationale: the review process worked — it caught everything, including fresh Criticals in Round 3. The defects should not have been there to catch. The recurring classes (cross-event IDOR, UTC-Kind, missing AC test files, audit placement) are all *checklist-able* — they are known patterns that a disciplined dev-story pass would have closed before review. Adding a mandatory mid-epic review treats the symptom; hardening the dev-story task lists treats the cause.

**Consequence for A10:** A10's judgment-only mid-epic trigger is **retired** (it didn't fire, and the shape was wrong twice now — A4 then A10). It is *replaced* by the dev-process checklists A12–A14, which are mechanical and gate `review` status rather than relying on developer judgment mid-flow. The hybrid CR+ER-at-boundary workflow otherwise continues unchanged — boundary review may still run multiple rounds; the goal of A12–A14 is to make Round 2 the last round.
