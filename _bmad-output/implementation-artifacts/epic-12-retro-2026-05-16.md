## 3. What Didn't Go Well — Patterns

*(Systemic observations, no blame. Each links to an action item.)*

- **A29 (AC-Subitem Check) worked for explicit-numbered ACs but NOT for implicit image-surface ACs.** E12-S1's spec did NOT enumerate "image contains no dev secrets" as a sub-AC (that wording existed in E12-S3 AC-14 for Keycloak only). So the dev-story Quality-Gates table had no row to fail. The Edge Case Hunter at review-time walked `COPY src/ src/` and surfaced `appsettings.Development.json` shipping with `dev-secret-change-me` / `admin-service-secret-2026` / `rustfsadmin/rustfsadmin` literals (P2). Pattern: **cross-story consistency for orthogonal AC dimensions (image surface, build-arg propagation, secret bake) needs explicit per-AC carry-forward.** → **A31** (Cross-Story Orthogonal-AC Inventory)
- **A30 `[!]` markers worked, but D3 reveals that `[!]`-deferred verifications can also block Code-Review decision resolution.** D3 (NEXTAUTH server-side `KEYCLOAK_ISSUER` vs build-arg `NEXT_PUBLIC_KEYCLOAK_ISSUER` divergence) is unverifiable without a browser smoke (Task 8 of E12-S4 is `[!]`). The review surfaced it as a Decision; resolution was "(A) verify-first via browser smoke". This is correct, but it means the Decision is **not closed** until human browser smoke runs — it's queued behind the existing `[!]` marker. Pattern: **Decision-Resolution options must include a "verify-first" branch when the falsifying evidence requires browser/manual interaction, AND the resolution must explicitly route into the existing `[!]` queue rather than getting silently dropped.** → **A32** (Decision-Resolution with Manual-Verify Hand-off)
- **`USER 1000` as a raw numeric UID without consulting the base-image's pre-created user was a spec-blindness issue.** E12-S1 spec said "non-root USER 1000" — a Kubernetes-style runAsNonRoot pattern. The implementation took it literally. But MS aspnet:10.0 since .NET 8 ships `USER $APP_UID` (1654) with a pre-created `app` user, `/home/app`, and `chown`-ed `/app`. Using `USER 1000` ran as a UID with no home dir, no /etc/passwd entry, and `/app` owned by root — breaking ASP.NET DataProtection writes silently. The fix in P1 used the base-image's user. **Pattern: when a Dockerfile spec prescribes USER semantics, the Task-0 spike must verify what user the upstream base image already provides.** → **A33** (Base-Image User Verification in Dockerfile Spikes)
- **D3 (NEXTAUTH issuer divergence) was discovered by review, not by Task 8.** The Task-8 manual-verify checkbox was `[!]`-deferred per A30. The Edge Case Hunter walked the env-var flow at review-time and found the divergence statically (server-side env vs build-arg). Without the adversarial review layer, the divergence would have shipped as a "works locally during overlay smoke" claim and surfaced as a "users get logged out after token refresh" bug in real Beta — exactly the failure mode A30 was supposed to surface. **Pattern: when an `[!]`-marked verification is for a complex multi-component interaction (NextAuth + Keycloak + overlay env wiring), a one-line static-analysis step in Quality-Gates ("walk the env-var flow from build-arg to runtime to token validation") would have surfaced D3 without needing a browser.** Linked to A31 (orthogonal-AC inventory must include "env-var propagation path consistency").
