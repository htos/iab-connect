## What went well

1. **A56 existing-implementation spike paid off across all three stories.** S1's cost-center CRUD, S2's whole association surface, and the transactions export were all confirmed already-shipping — turning what the raw ACs read as "build" into clean verification. Zero accidental rewrites.
2. **The E4/E5 action items (A61–A69) prevented the usual epic-boundary defects.** This is the first feature epic since the pivot to close with **0 boundary-review patches**. A63 (no new injected services → no harness breakage), A64 (stable `useTranslations`), A58 (scoped lint/format), and the single-aggregate handler shape all applied without friction.
3. **DEC-1's entity-reuse decision (ActivityArea = cost center)** kept the data model clean — one allocation dimension on finance records, not two overlapping ones — and shrank the epic from "3 build-stories" to "1 build + 1 verify + 1 report".
4. **Honest scoping (A65/A68).** S2 plainly stated it was mostly verification; S3's QGT stated the concrete v1 actuals behavior (SimpleCash `Transaction`) rather than the aspirational "all finance records".

## What was tricky / lessons

1. **Actuals source is accounting-mode-dependent, and a report that silently shows 0 outside its mode is a trap.** S3 sums `Transaction` (correct for the default/Beta SimpleCash mode); a DoubleEntry installation would see actuals = 0 with no visible reason. Disclosed as a v1 boundary (E6-FT-1) and correct for the Beta, but it surfaced the general rule below (A71).
2. **A pre-check + DB-unique-index pair returns two different HTTP codes** (409 from the pre-check, 500 from the index on a concurrent race). The architecture boundary (no EF in Application) makes the clean fix belong at the repository layer (E6-FT-2).
3. **A consumed sibling-story surface must be checked on the read side too (A62).** The journal-line selector only round-trips on edit because `JournalEntryLineDto` already exposed `ActivityAreaId` — verified rather than assumed.

## New action items (promoted to project-context.md)

- **A70 — Reuse an existing tagging/dimension entity for a new allocation concept instead of minting a parallel entity.** When a new planning/allocation concept (cost center, category, bucket) maps onto an entity that already exists and is already referenced by the target records, reuse it; do not add a parallel aggregate that creates two overlapping allocation dimensions on the same records. Decide this at create-story/architecture time. Pattern: E6-S1 DEC-1 made `ActivityArea` (REQ-068) the cost center — `Transaction`/`InvoiceItem`/`JournalEntryLine` already carried `ActivityAreaId`, so a new `CostCenter` entity would have duplicated the dimension and forced a double-allocation choice on every finance write.
- **A71 — A report whose source is mode/config-dependent must declare its covered mode and degrade visibly, not silently to zero (refines A68).** When a read-only report's data source depends on a runtime mode/config (accounting mode, feature flag, module state), the report must either cover all relevant modes or visibly state which mode it covers; silently returning 0 outside the covered mode reads as "nothing happened" (doing-wrong), not "less". A68 requires the QGT to state the concrete behavior; A71 extends it to the *user-facing surface* for config-scoped reports. Pattern: E6-S3's Soll/Ist sums `Transaction` (SimpleCash); a DoubleEntry installation sees actuals = 0 with no on-screen cue (E6-FT-1).
