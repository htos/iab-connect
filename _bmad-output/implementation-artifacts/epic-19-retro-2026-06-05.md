## What to improve (→ action items)

The epic-boundary review's three §8 patches (P1/P2/P3) share one root cause: **the custom-domain section enumerated the *named parity anchors* but missed the *other consumers of the same changing host*.** The five-anchor issuer parity was front-of-mind (it has a name + a docs/14 script), so `web.KEYCLOAK_ISSUER` made the parity table but not the actionable checklist, while `NEXTAUTH_URL` and `FRONTEND_PUBLIC_URL` — host consumers that aren't part of the *issuer* parity set — were missed entirely. A "host change" runbook needs to sweep **every** per-service variable that embeds the changing host, not just the famous invariant.

P4 is a narrower lesson: doc-vs-code accuracy extends to **library capability**, not just binary presence — the app's `System.Net.Mail.SmtpClient` is STARTTLS-only, which constrains a documented port choice.

## New action items (promoted to project-context.md)

- **A59 — Host/URL-change runbooks must sweep every consumer var across every service, not just the named invariant.** When a runbook documents changing a host / domain / base-URL / issuer, enumerate **every** Railway variable (across all services) that embeds that host by reading the per-service variable tables (docs/14 §5.1–§5.3) — not only the famous parity set. Surfaced in E19-S1 boundary review P1/P2/P3: the five-anchor *issuer* parity was covered, but `web.NEXTAUTH_URL` (web's own public URL) + `keycloak.FRONTEND_PUBLIC_URL` (second realm redirect URI) + the runtime side of `web.KEYCLOAK_ISSUER` were host consumers outside the named invariant and were missed. Generalises A40/A52: verify the *full consumer set* of a changing value, not just its headline invariant.
- **A60 — Doc-vs-code accuracy covers library/transport capability, not just binary presence (refines A45/A57).** A45/A57 check that a documented *binary* is reachable in the runtime image. Extend to **library capability**: when a doc prescribes a protocol/port/option, confirm the application's actual client library supports it. Surfaced in E19-S4 boundary review P4: the doc offered SMTP port 465 (implicit TLS), but the app's `System.Net.Mail.SmtpClient` does STARTTLS only and does not reliably support implicit-TLS/SMTPS on 465. The A42 reread must cross-check documented protocol options against the consuming library's documented capabilities.

A28–A58 carry forward.
