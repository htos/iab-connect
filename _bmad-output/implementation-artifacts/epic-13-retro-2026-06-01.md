### 2.2 Adversarial epic-boundary code review caught 3 structurally-critical bugs the dev-agent could not have caught alone

The Edge Case Hunter — with full project read access — identified three issues that would have killed the first deploy on Harry's session:

- **E1**: `iabconnect-api` is `bearerOnly: true` in the realm JSON (line 242). `Keycloak__ClientSecret` documented in Section 5.1 + rotation runbook in Section 7 was a **structurally unused secret** — `grep -r 'Keycloak__ClientSecret'` over `backend/src` returns 0 hits. The dev-agent had carried the secret-spec from the story file's AC-2 text without cross-checking the realm JSON. Removing it from §5.1, §5.3, §7 in patch P1 prevented a 30-minute "why isn't the Credentials tab showing for `iabconnect-api`?" diagnostic loop.
- **E2**: Realm JSON uses `${IABCONNECT_BETA_HOST}` and `${FRONTEND_PUBLIC_URL}` as Keycloak realm-import placeholders for `iabconnect-frontend.redirectUris` / `webOrigins` (realm:257-262). Section 5.3 documented only the three `IABCONNECT_*_CLIENT_SECRET` placeholders, missing both `_HOST` / `_URL`. Without them set, Keycloak substitutes empty strings → `redirectUris=["/*", "/*"]` → first OIDC callback fails with `Invalid parameter: redirect_uri` → Section 10.4 step 4 browser smoke breaks with no api-side log. Patch P2 added both vars with explicit scheme-required rationale.
- **E3**: `KEYCLOAK_ISSUER` parity check covered 3 anchors but missed `keycloak.KC_HOSTNAME` (Section 5.3) and `api.KeycloakAdmin__BaseUrl` (Section 5.1). Patch P3 expanded to a 5-anchor diff with concrete shell snippet and called out the "https:// prefixed onto KC_HOSTNAME → resulting issuer URL becomes `https://https://...`" double-scheme footgun that breaks everything downstream silently.

None of these were "obvious from the story spec"; all required cross-referencing the realm JSON + backend code against the doc. The hybrid CR+ER-at-epic-boundary policy (project memory `feedback_bmad_workflow`) prevented Harry from hitting these live on Railway.

### 3.2 Three structural Keycloak gaps (E1/E2/E3) shipped because the dev-agent didn't cross-check realm JSON

### 3.3 The doc shipped with three categories of staleness the Blind Hunter caught

- File:line anchors had drifted (`Keycloak__Authority` cited :121, actual :139; `Keycloak__ClientId` cited :122, actual :140). The dev-agent inherited the line citations from the older story-file Task notes without re-verifying against current HEAD.

### 3.4 `kc.sh bootstrap-admin` recovery procedure was wrong (Keycloak 26 syntax + railway run semantics)

§11.2 recovery for "Keycloak admin can't sign in" prescribed:

```sh
railway run --service keycloak kc.sh bootstrap-admin --realm master --username recovery --password '<new-strong-random>'
```

Three things wrong:
1. Keycloak 26 subcommand is `bootstrap-admin user` (the `user` subcommand) not bare `bootstrap-admin`.
2. There is no `--realm` flag — `bootstrap-admin user` always targets `master`.
3. `railway run` evaluates the command on the **local** machine with the service's env-vars injected, not inside the container. `kc.sh` is only on `$PATH` inside the keycloak container; the recovery command would fail at `bash: kc.sh: command not found`.

The dev-agent invented the syntax plausibly rather than verifying against Keycloak 26 docs. Patch P11 corrected to the actual subcommand + `--password:env` + Railway dashboard "Open shell" semantics.

**Lesson:** when documenting recovery procedures for tools the dev-agent doesn't directly exercise in the session, the command is unverified. Either verify against the tool's docs OR mark the procedure `[!] verify against Keycloak 26 docs before executing`.

### 3.5 Section 8.3 negative-test commands had a silent-success failure mode

The "private services should be unreachable" verification block ran `timeout 10 psql -h postgres-app.railway.internal ...`. `psql -U postgres` without `-w` hangs on the interactive password prompt for the full timeout budget — outwardly indistinguishable from a successful "host unreachable" timeout. Patch P13 added `dig +short` as the primary DNS-failure proof + `psql -w` + `PGPASSWORD=irrelevant` to suppress the prompt.

This is a class of bug the dev-agent has hit before (E12 retro mentioned shell-command edge cases in the docker-compose smoke). Verification commands that look obviously right can have subtle interactive-stdin failure modes. **Lesson:** for shell commands that involve interactive tools (psql, ssh, sudo, gh), explicitly add the non-interactive flag (`-w`, `-o BatchMode=yes`, `-n`, etc.).

### 3.6 Doc-bundle pattern leaked a sprint-status note into operational doc

Section 9.1 carried "E17-S3 (originally Wave 8) is absorbed into E13-S4 as a result; sprint-status flips e17-s3 to `done` with a `done-absorbed-by-e13-s4` note." — operational scope leak. A self-hoster following the doc has zero need to know BMad sprint history. The Blind Hunter caught it; P18 removed it. **Generalizable:** the doc-bundle pattern (§2.1) is great for cohesion but creates a temptation to mix internal sprint-tracking commentary into operator-facing content. Filter aggressively at story-close.

## 5. New action items from E13 retrospective

### A38 — Doc-bundle pattern: when N stories all ship into the same single output doc, write the complete doc once in the first story

When the user requests multiple stories whose only repo-tracked artifacts are sections of one logical document (Beta deployment checklist, security runbook, migration guide, etc.), the dev-agent should write the **complete document** in the first story rather than authoring an initial cut and *appending* in subsequent stories. Subsequent stories close by pointing their Quality-Gates evidence column at the relevant section anchors, not by appending.

**Reason:** appending in N stories produces N section-numbering inconsistency risks, N lint-style drift opportunities, N Markdown-link-anchor checks, and N "is the section I'm appending under really the right place?" moments. Writing once and pointing back is strictly cleaner.

**How to apply:** at the start of a multi-story burst where each story's File List includes the same doc, the dev-agent recognizes the cohort, writes the doc once in the first story with stub sections for the other stories' content, and each subsequent story's Dev Agent Record points its Quality-Gates anchors at the pre-existing sections. The first story's File List lists the doc as NEW; subsequent stories' File List shows the doc as "covered by [first-story]'s creation" with a note.

**Anti-pattern to avoid:** don't apply this when the doc is genuinely topic-bounded per story (a per-feature design doc, a per-component API reference). The pattern works when the doc is **operator-facing** and the stories carve up its sections.

### A39 — When documenting Keycloak env vars, grep the realm JSON for ALL `${...}` placeholders + `"bearerOnly": true` clients

Generalization of E1/E2/E3 review patches. When a story documents Keycloak service env vars, the dev-agent MUST:

1. `grep -oP '\$\{[A-Z_]+\}' infra/keycloak/realms-beta/iabconnect-realm.json | sort -u` → enumerate every realm-import substitution placeholder. Every one needs a producer-side env-var row in the doc.
2. `grep -B 5 '"bearerOnly": true' infra/keycloak/realms-beta/iabconnect-realm.json` → identify every bearer-only client. Bearer-only clients do NOT have secrets; their consumer-side env-var rows must not include `*_ClientSecret`.

**Reason:** E13-S2 documented three `_CLIENT_SECRET` placeholders but missed `${IABCONNECT_BETA_HOST}` + `${FRONTEND_PUBLIC_URL}` → first browser smoke would fail with `Invalid parameter: redirect_uri`. Separately, the dev-agent prescribed `Keycloak__ClientSecret` for the bearer-only `iabconnect-api` client → secret would be Sealed + rotated + worried-about, all for a value nothing reads.

**How to apply:** add a "realm JSON cross-reference" Task 0 sub-step to any Keycloak-env-var-touching story spec.

### A40 — Verify shell-command syntax for tools the dev-agent doesn't exercise in-session before documenting recovery procedures

Generalization of patch P11 (`kc.sh bootstrap-admin` wrong subcommand + `railway run` semantics misdescribed). When the doc prescribes a recovery command for a tool the dev-agent has not directly exercised in the session (Keycloak CLI, Railway CLI, Postgres CLI in non-trivial modes, Docker swarm operations), the command must be either:

- verified against the tool's current docs (cite the doc anchor), OR
- explicitly marked `[!] verify against <tool> docs before executing` so the operator knows it's plausible-but-untested.

**Reason:** invented-but-plausible recovery commands are worse than no recovery procedure — they consume operator time during a real incident before the operator discovers the command doesn't exist.

**How to apply:** at story-authoring time, the dev-agent classifies each shell command in the doc by whether it was exercised in-session, and either verifies the others or marks them `[!]`. Story Quality-Gates AC for "recovery procedures documented" requires one of the two states.

### A41 — A32 escape hatch for "pre-declared autonomous mode" (or rule that AskUserQuestion is always required)

The DEC-1 auto-resolution in E13-S4 ran into the tension between A32 (AskUserQuestion at Task 0 for Decision-Needed) and the user's explicit session-level "no stopping just straight forwards, implement them full" directive + project memory `feedback_decisions_via_ask_tool` ("always use AskUserQuestion for choice prompts"). The dev-agent picked the user's no-stopping signal as more session-current and auto-resolved Option A with explicit rationale tied to the user's words.

The Acceptance Auditor flagged this as defensible-but-worth-clarifying. Two paths forward:

**Path A: A32 always requires AskUserQuestion.** The user's no-stopping directive does NOT override A32; the dev-agent must surface the AskUserQuestion once even in autonomous mode, and the user typing a one-letter answer is "not stopping" for the purposes of the directive. Project memory `feedback_decisions_via_ask_tool` aligns with this.

**Path B: A32 has an explicit "pre-declared autonomous mode" escape clause.** When the user pre-declares autonomous-mode for a session AND the story file recommends one of the options AND the dev-agent records the recommendation + reasoning in the Debug Log per A32 (a)/(b)/(c), step (d) AskUserQuestion is skipped. The escape clause requires all three preconditions.

**Recommendation:** Path B with explicit precondition logging. The user's autonomous-mode directive should not be silently overridden by a process rule; equally, A32's intent (the user knowingly makes the decision) is preserved as long as the dev-agent's auto-resolution maps to the story-file recommendation + cites the user's directive.

**How to apply:** if user picks Path B, add the escape-clause language to A32 in project-context.md. If Path A, remove the auto-resolution and re-do DEC-1 with proper AskUserQuestion as a single-question retro action (no implementation impact — Option A was the right answer, just procedurally surfaced).

→ **Decision deferred to Harry.** Logged at deferred-work.md E13-FT-10 already; the retro is the appropriate place to resolve.

### A42 — "Reread the doc as a stranger" is the documentation-equivalent of code review

When a story's deliverable is a doc (especially an operator runbook), the AC-Subitem Completion Check (A29) is necessary but not sufficient — A29 verifies each AC's literal completion, not whether the doc reads coherently end-to-end. The Blind Hunter's 17 findings on E13's doc included:

- Internal contradictions between adjacent sentences (README "immutable images + moving tag").
- Pre-checked status boxes (turning a reusable checklist into a one-deploy snapshot).
- Stale file:line anchors carried from older story drafts.
- A "5-line route" claim contradicting the 8-line file.
- Scope-leak (sprint history bleeding into operator-facing content).

None of these violate any specific AC — they violate the **spirit** of "doc is reproducible end-to-end". A29 misses them by design.

**How to apply:** for any story whose primary deliverable is documentation, the dev-agent runs a fresh-eyes re-read pass before flipping to `review`, looking specifically for: cross-section contradictions, pre-filled placeholders that should be blank, stale anchor citations, scope-leak from internal-process notes. This is what the Blind Hunter does at epic boundary — pulling it forward to story close means epic-boundary review catches the structural issues (which the Edge Case Hunter found) without burning cycles on surface-level slips.
