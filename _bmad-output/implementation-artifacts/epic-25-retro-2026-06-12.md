## What the boundary review caught (the durable lessons)

The net was green + tsc + build all passed, yet the adversarial per-slice review found **9 behaviour-preservation divergences** the net didn't cover. Four generalize into new action items:

### A95 — RHF+Zod migration of a manual `<select>` form must round-trip an out-of-set stored value (HIGH; from E25-S2 boundary P1)

When a manual form holds a `<select value={state}>` where `state` is the RAW stored value, a controlled select whose value matches no `<option>` **leaves the submitted value unchanged** — so an out-of-set value (e.g. a `segmentType`/`consentFilter` the backend accepts but the form only renders a subset of) round-trips on a no-touch save. Migrating that to RHF+Zod with `z.enum(renderedSubset)` **breaks this**: the schema rejects/coerces the out-of-set default, and a native `<select>` with no matching option reports `.value===""`, so a no-touch edit-save silently REWRITES the field. **Rule:** when migrating such a form, (1) widen the Zod field to the FULL transport union (not the rendered option subset), (2) keep the RAW `initial` value in `defaultValues` (no `includes`-coercion), and (3) render the out-of-set initial value as an extra `<option>` so the native select displays + round-trips it. Add a regression test asserting a no-touch edit-save emits the original out-of-set value. Refines A92 (error-path input preservation) to "option-subset value preservation on edit."

### A96 — never use Zod `.trim()` (or any transform) on a field whose submitted bytes must match the god-page (MED; recurred in E25-S3 P4 AND E25-S4 P6)

`z.string().trim()` TRANSFORMS the resolved value, so RHF's `handleSubmit` passes the TRIMMED value to `onSubmit` → the POST/`onSave` payload is no longer byte-identical to a god-page that sent the raw input (and it rejects whitespace-only that HTML5 `required` accepted). This recurred in TWO slices in one epic. **Rule:** in a behaviour-preserving form migration, validate emptiness with `z.string().min(1, key)` or `.refine(s => s.trim().length > 0, key)` — NEVER a mutating `.trim()`/transform — on any field whose submitted bytes must match HEAD. Companion: an RHF+Zod form using `<form noValidate>` MUST render the per-field Zod error messages (the sponsor-form recipe), else it silently drops the god-page's native-`required` feedback (E25-S3 P4). Extend the documented Form Sub-Recipe with both points.

### A97 — a migrated query's `enabled` gate must match the god-page's FETCH gate, not the page's render/redirect role-gate (MED; from E25-S2 P2; refines A90)

A god-page detail/edit that fetched on TOKEN-only (no client role check) surfaced a backend 403 as a graceful error panel. Migrating the fetch to a TanStack query gated `enabled` on the page's ROLE (`isVorstand || isAdmin`) means a direct-nav user who lacks the role never runs the query → `data` undefined + `isError` false → a permanent stuck spinner (the error surface is lost). **Rule:** set the query `enabled` to the god-page's actual FETCH precondition (usually `isAuthenticated && !!token`), NOT the page's render/redirect role-gate; keep the role check as the ACTION/edit-button gate. Verify the `isError && !data` branch renders the error/forbidden surface. A90 covered the list/load-effect early-return surface; A97 covers the detail/edit `enabled`-gate granularity. (List pages still role-redirect; detail/edit pages fetch-then-403, per the god-page.)

### A98 — a shared create/edit form's mode-divergent surfaces must be threaded through props AND pinned PER MODE in the net (HIGH; from E25-S3 P3; extends A80/A65)

A single shared `<Form mode="create"|"edit">` component that HARDCODES a mode-divergent surface (here the header back-link `href` + label — edit must point to the detail page + singular label, create to the list + plural) silently regresses the un-pinned mode. The characterization net missed it because it only pinned the create + detail back-links, not the EDIT one. **Rule:** thread EVERY mode-divergent surface (back-link href+label, submit label, post-submit redirect target, mode-specific guards) through props (mirror how `cancelHref` is already threaded), and the characterization net MUST pin both modes' divergent surfaces — a green net that only pins one mode is a false oracle for the other. Extends A80 (per-surface coverage) and A65 (a multi-surface AC is a coverage trap) to the create-vs-edit mode axis of a shared form.

### A99 (refines A93) — the deterministic-404 sentinel's feasibility is a function of the transport's status surface

A93 says disable retry for a deterministic-404 sentinel. Whether a REAL sentinel is feasible depends on whether the transport surfaces an HTTP status: `useApiClient` returns `{status}` and the legacy `ApiClient` throws `ApiError{statusCode}` → a real `XNotFoundError` on 404 + retry-exclusion (email-campaigns, email-templates); a token-fn that throws a generic `Error("loadError")` with no status (automations `@/lib/api/automations`) cannot distinguish 404 → use `retry:false` to preserve the god-page's single-fetch-then-error (the sentinel class may still be defined for parity, but it never fires). **Rule:** at create-story/dev time, read the wrapped/built transport's error surface and pick real-sentinel vs `retry:false` accordingly; record which in the DEC notes.
