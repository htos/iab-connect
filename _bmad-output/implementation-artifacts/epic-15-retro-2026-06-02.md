## §6. New action items (A43–A45)

### A43 — "(a)/(b)/(c) Debug Log template" for A32/A41 Decision resolutions

When a story has a DEC-Needed block and the user has pre-declared autonomous mode (A41), the dev-agent records the resolution in Dev Agent Record → Debug Log References using **exactly** this three-bullet shape:

- (a) **Option chosen**: `<the option letter + a 1-line summary>`
- (b) **Rationale**: `<at least three short sub-claims: (1) story recommendation; (2) user autonomous-mode verbatim quote; (3) downstream architectural justification>`
- (c) **Consequence chain**: `<which ACs / sub-items flip in scope or out of scope as a result>`

**Why:** E15 exercised A41 across two DEC-1s; both Dev Agent Records independently converged on this three-bullet structure but with slightly different ordering + level of detail. Pinning the template makes future audits + future agents reading these as examples consistent. Also helps when a Decision is later challenged in retro — the (a)/(b)/(c) record is a structured deposition.

**How to apply:** any future `bmad-dev-story` workflow run for a story containing a DEC-Needed block, when A41 preconditions hold, formats the Debug Log entry per A43.

### A44 — Hangfire recurring-job-id global-uniqueness test

When a new Hangfire recurring job is registered in `Api/DependencyInjection.cs`, the registering story MUST add (or extend) a test that asserts the full set of project-wide job-ids is distinct.

**Why:** Hangfire's `AddOrUpdate` is keyed by id; two registrations sharing an id silently overwrite each other in the recurringjob storage table. Without this guard, a copy-paste regression (the most likely failure mode when adding a 7th job by templating from one of the existing 6) is detected only at the first 03:00 UTC deploy — by which point the orphaned schedule's data damage may have begun.

**How to apply:** the test `RecurringJobIds_AreGloballyUnique_AcrossAllRegisteredJobs` in `RegisterDailyBackupJobTests.cs:54` is the canonical anchor. Future stories adding a new job-id constant extend the `ids` array + bump the `.Should().HaveCount(N)` assertion.

### A45 — A42 sixth category: documented-binary-surface reachability

When a documentation deliverable includes a shell command that invokes a non-trivial binary (CLI tool, container utility, language interpreter), the A42 reread-as-a-stranger pass MUST additionally confirm:

- **The binary is reachable from the documented invocation site** — verify by reading the relevant Dockerfile / installation script / image-build pipeline. If the documented context is "run from inside the api container" but the Dockerfile does not install the binary, the snippet is invalid as-shipped.
- **The binary version matches the spec contract** — if the doc says "uses postgresql-client-17 features", verify the Dockerfile installs v17 specifically.
- **If the binary is operator-provided (must install locally), the doc says so explicitly** — not silently assumed.

**Why:** E15-S3 boundary review P2 caught Section 15.4 + 15.5 Step 1 prescribing `aws s3` commands against a backend container that does not install the AWS CLI. A42's existing five categories (cross-section contradictions / pre-filled status / stale anchors / imprecise claims / sprint-tracking leakage) cover doc-internal consistency but not doc-vs-implementation-image reachability. A45 extends A42 to cover this surface.

**How to apply:** the dev-agent's A42 pass for any story shipping doc content with shell commands runs through this sixth checklist item before flipping to `review`.
