## §4. New action items (A46–A48 candidates)

Three new patterns surfaced in this epic worth promoting to project-context.md as durable rules:

### A46 — A35 refinement: `afterEach(cleanup)` applies to tests that call Testing-Library `render()`, NOT all `*.test.tsx` / `*.test.ts` files

**Surfaced by:** E16-S1 (`dockerfile-public-vars.test.ts` — file-read + regex assertions, no DOM) + E16-S2 (`auth.logout.test.ts` — mocked-module + window patching, no DOM) both implementing without `afterEach(cleanup)`. Original A35 wording said "every new `*.test.tsx` that uses Testing Library's `render()` must include `import { cleanup }` + `afterEach(cleanup)`" — but this got over-broadly applied in the story spec ("include `afterEach(cleanup)` even if not rendering (consistency with other suites)").

**Why:** `cleanup` unmounts the React tree mounted by `render()`. Without a `render()` call, there is nothing to unmount; calling `cleanup` is a no-op but the `@testing-library/react` import has a non-trivial cost at test-collection time (it pulls jsdom + React-DOM into the file graph). For pure-Node tests (file IO, regex, mock assertions), the unnecessary jsdom pull also forces the `// @vitest-environment jsdom` directive unnecessarily.

**How to apply:**
- A test that calls `render(...)` from `@testing-library/react`: include `import { cleanup } from "@testing-library/react"; afterEach(cleanup);` + the `// @vitest-environment jsdom` directive at file top.
- A test that does NOT call `render(...)`: do NOT import cleanup, do NOT add the directive unless other browser-API access (e.g., `window.location`) is required. The `auth.logout.test.ts` does need jsdom (for `window.location.origin` patching) but does NOT need cleanup.

**Refinement of:** A35 (Epic-20 retro 2026-06-01).

### A47 — Autonomous-mode (A41 path B) escape extends to the live-walkthrough `[!]` queue, not just DEC-Needed AskUserQuestion

**Surfaced by:** E16 — three stories' worth of live-Beta walkthrough ACs (~30 items) were uniformly flipped to `deferred-pending-beta-green` in the Quality-Gates Closing tables. This is a broader application than A41's original DEC-Needed scope, but the three preconditions (user pre-declared autonomous mode + story file documents the deferral as designed + dev-agent records (a)/(b)/(c)) still hold.

**Why:** The dev-agent cannot orchestrate live Beta deploys + browser sessions + Railway dashboards from its sandbox. When the user has explicitly pre-declared that all live-Beta `[!]` items will be batched into a single end-of-epic-set session, individually surfacing each `[!]` for AskUserQuestion is theatre — it wastes context budget and produces a single uniform answer.

**How to apply:** When ALL THREE preconditions hold for a live-walkthrough `[!]` queue (not a DEC-Needed), the dev-agent may flip the queue uniformly to `deferred-pending-beta-green` without per-item AskUserQuestion:
1. The user has pre-declared the deferral in this session (e.g., "i will do everthing at the end once im finished with all epic. then i need your support.").
2. Each story file's design pre-acknowledges the live-Beta gate as `[!]` Task 0 prerequisite + paste-blanks in the doc-bundle deliverable.
3. The dev-agent's Dev Agent Record carries a single (a)/(b)/(c) block PER STORY that quotes the user's autonomous-mode verbatim directive and enumerates the row-flips in the QG table as consequence chain.

The Quality-Gates Closing table must clearly distinguish `covered` vs. `deferred-pending-beta-green` per row — no aggregate claims. The unified human-verify queue (the "Q1..QN" table in each story's Completion Notes) is the operator-facing artifact that surfaces the deferred items at execution time.

**Refines:** A41 (Epic-13 retro 2026-06-01) — A41's original scope was DEC-Needed at Task 0; A47 extends to the broader `[!]` live-walkthrough queue when the same preconditions hold.

### A48 — Testcontainers package tags + image digest pinning

**Surfaced by:** E16-S3 introducing `Testcontainers.Minio` 4.10.0 + the `MinioBuilder("minio/minio:RELEASE.2024-12-13T22-19-12Z")` constructor (a tag pin, NOT a digest pin). The earlier `Testcontainers.PostgreSql` usage in `Infrastructure.Tests` similarly uses tag pins via implicit defaults.

**Why:** Tag pins are mutable — `minio/minio:RELEASE.2024-12-13T22-19-12Z` could in principle be re-tagged to a different image content if the upstream registry allows it. Reproducible test runs benefit from digest pins (`@sha256:<digest>`) at least for CI. ADR-014 explicitly chose digest pins for production images; the same hygiene should apply to test-container images.

**How to apply:** When adding a new `Testcontainers.<X>` package or pinning a new container image tag in a test fixture, document the spike-time digest of the chosen tag in a code comment + plan a follow-up to convert to `<image>@sha256:<digest>` form. For Testcontainers 4.10.0 the `WithImage("image@sha256:...")` form is supported.

**Spike-discovery target:** the actual SHA-256 digest of `minio/minio:RELEASE.2024-12-13T22-19-12Z` at the time of this commit's first publish. Open question whether the digest pin is in-scope for E16 epic-close patch or as a future-work item.

**Deferred-work entry candidate:** E16-FT-1 — pin `Testcontainers.Minio` MinioBuilder image to its digest at `S3DocumentStorageIntegrationTests.cs:31`.

5. **A45 (documented-binary-surface reachability)** surfaced a strict application in Section 19.3 Path B (`railway shell -s rustfs --command 'which mc'` reachability gate before any mc usage). This is the strongest A45 application to date and worth highlighting as a pattern for future operator-doc sections.
