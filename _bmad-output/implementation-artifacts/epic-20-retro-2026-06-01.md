## 3. What didn't go well

### 3.1 LicenseFooter test failed initially due to missing `cleanup()` between renders

`LicenseFooter.test.tsx` initial run produced 4 `getMultipleElementsFoundError` failures because each `render()` accumulated DOM nodes across tests. Project's Vitest setup doesn't auto-cleanup; the fix (explicit `cleanup()` in `afterEach`) is one line. This is a project-specific pattern the dev-agent didn't know — would have been caught faster with an explicit "frontend Vitest setup quirks" entry in project-context.

→ **New action item A35.**

### 3.2 `About_UnknownWhenEnvVarsMissing` integration test was environment-dependent

Test asserted `commitSha == "unknown"` by relying on host env not exporting `BUILD_SHA`. Real risk in CI runners. The unit test (`AboutResponseBuilderTests`) was correctly environment-independent (uses `ConfigurationBuilder` with `AddInMemoryCollection` only); the integration test was not. The Edge Case Hunter caught it; the patch (P4) explicitly binds empty values in the test factory's `AddInMemoryCollection`. A general lesson for ASP.NET Core integration tests reading env-var-mapped config.

→ **New action item A36.**

### 3.3 `dcoapp/action` referenced by spec didn't exist at impl time

E20-S1 spec said "Recommended action: `dcoapp/action`". At implementation time `https://github.com/dcoapp/action` returned HTTP 404 (repo deleted/moved). The dev-agent substituted an inline `git rev-list` + `grep` bash script — cleaner outcome (zero third-party action dependency to pin) but introduced 2 of the code-review defers (D1 fork-PR base-fetch, D2 CRLF tolerance, D3 trailer-email-vs-author).

→ The story spec should not have hardcoded a specific GitHub Marketplace action without a freshness check. **General lesson on third-party-action references in specs:** add a "verify-at-impl-time" note for any GitHub Marketplace action by name, AND list 1-2 alternates.

### 3.4 README "License" section was a pre-existing contradiction E20-S1 missed

E20-S1 AC-5 was scoped narrowly to the badge swap at line 25. The README's actual "License" section (line 937) still said "**proprietary software** ... Unauthorized copying, modification, distribution, or use ... strictly prohibited" — directly contradicting the new AGPL LICENSE file. E20-S1 shipped the contradiction. E20-S2 caught it (as a "scope addition") and fixed it. The acceptance-auditor would have flagged a public-OSS repo with a license-contradiction in its primary marketing page as a release blocker.

→ **General lesson:** when a story changes a project's license, the dev-agent should grep the README for license-related text and fold any contradicting prose into the same story. The narrow AC scope was correct on the strict reading but missed the spirit of "publish AGPL-3.0-or-later cleanly".

### 3.5 Continuous-session delivery burned cycles on full-test-suite re-runs

Full backend `dotnet test` runs took ~3 minutes each. I ran it 3 times in this session (post-E20-S3, post-E20-S4 unrelated, post-review-patches). The middle one was unnecessary. **Procedural lesson:** for cross-cutting story batches, only re-run the full suite after the dev work on the batch is complete + after any review patches are applied. Story-scoped filtered runs (`--filter "FullyQualifiedName~..."`) are sufficient between batches.

## 5. New action items from E20 retrospective

### A34 — Bulk spec-refresh at epic start when implementing multiple stories in one session

When the user requests a multi-story burst ("implement the whole epic"), the create-story refresh pass on each story should happen as a single front-loaded batch BEFORE any dev work starts. The e20-s3 refresh was done one-off; for s1/s2/s4/s5 I refreshed implicitly via the bulk-2026-05-15 story file already-existing. A single batch-refresh would have caught the dcoapp/action 404 (e20-s1) and the README license-contradiction (e20-s2) before dev started.

**Reason:** late-discovered spec-vs-reality drift mid-implementation requires either a deviation note (acceptable but adds review burden) or a mid-flight task addition (breaks the "one story = one implementation arc" mental model).
**How to apply:** when the user requests N≥3 stories in one session, dev-agent runs N create-story refreshes upfront, surfaces all spec-vs-reality deltas as a batch, gets user sign-off on the deltas, then proceeds to dev-story for each.

### A35 — Frontend Vitest + Testing Library: `afterEach(cleanup)` is mandatory in this project

Vitest in this project does NOT auto-cleanup between tests. Each `render(<Component />)` accumulates DOM in the shared `document.body`. Tests using `screen.getByText(...)` will fail with `getMultipleElementsFoundError` from the second test onwards.

**Reason:** the `LicenseFooter.test.tsx` initial run failed 4 of 6 tests with this exact error; one-line fix to add `import { cleanup }` + `afterEach(cleanup)`.
**How to apply:** every new `*.test.tsx` that uses Testing Library's `render()` must include `cleanup()` in `afterEach`. Promote to `project-context.md` under "Testing Rules".

### A36 — ASP.NET Core integration tests reading env-var-mapped IConfiguration MUST override in TestWebApplicationFactory

ASP.NET Core's default `IConfiguration` chain reads unprefixed env vars (e.g., `BUILD_SHA`, `BUILD_DATE`, `PATH`, `HOME`, anything else). A test asserting `configuration["FOO"]` returns null is environment-dependent. Fix: in `TestWebApplicationFactory.ConfigureAppConfiguration`, explicitly bind empty values for any env-var-mapped key the test reads — `AddInMemoryCollection` runs LATER than the env-var provider so it overrides.

**Reason:** `About_UnknownWhenEnvVarsMissing` would have flaked the moment CI exported `BUILD_SHA` (which the new E20-S5 GHCR workflow does). P4 patch fixed it.
**How to apply:** any new endpoint test that asserts the absence of an env-var-mapped config key must extend the TestWebApplicationFactory InMemory dictionary. Promote to `project-context.md` under "Testing Rules".

### A37 — License-related changes must grep README for contradictory prose

When a story changes the project's license (LICENSE file, badge, SPDX identifier, etc.), the dev-agent must grep README + top-level docs for license-related text and fold any contradicting prose into the same story.

**Reason:** E20-S1 shipped the AGPL LICENSE file while README still claimed "**proprietary software** ... All Rights Reserved". E20-S2 had to fix it as scope addition. A public-OSS repo with a license-contradiction in its primary marketing page is a release blocker.
**How to apply:** add a "license-related grep" subtask to any future license-touching story. Grep patterns: `proprietary|All Rights Reserved|Private` (red flags); `License|Licensed|MIT|GPL|Apache` (review for accuracy).
