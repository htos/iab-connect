## 2. What worked

1. **Three-layer adversarial review caught issues no single reviewer would have.**
   The Blind Hunter (diff-only Cynical Review) surfaced the LIKE-wildcard injection (P1) and the Keycloak-orphan policy gap (D4) by being naïvely suspicious. The Edge Case Hunter found the EF SaveChanges-ordering issue around the unique partial Keycloak index (P6) by walking the code path with project context. The Acceptance Auditor confirmed all 32 ACs met. Each layer caught a different class of issue.

2. **Repository-level race-recovery preserved Clean Architecture.**
   The concurrent-dismissal race (P4) initially leaked `Microsoft.EntityFrameworkCore.DbUpdateException` into the Application layer — which fails the build because Application doesn't reference EF Core. Refactoring into `IDuplicateCandidateDismissalRepository.AddAtomicAsync` returning `(entity, created)` kept the Clean-Arch boundary intact and moved the EF concern to Infrastructure where it belongs. Future race-recovery patterns can reuse this shape.

3. **Action A2 (Symmetric-Guard Checklist) earned its keep twice.**
   E2.S2 added the email-normalization bullet ("when you migrate one call site to a normalized lookup, audit every other call site for the same email field"). E2.S3 added the soft-retire-flag bullet ("when introducing `MergedIntoMemberId`, every read query whose semantics rely on 'active members' must filter the flag"). Both bullets were applied in their own stories AND prevented the boundary review from finding asymmetric exposure regressions.

4. **EF migration tooling held.**
   `dotnet ef migrations add ChangeMergedIntoMemberFKToRestrict` ran cleanly post-patch; the empty Up/Down (because the snapshot was hand-pre-updated) was filled in manually with explicit `DropForeignKey` + `AddForeignKey` calls. Migration applies cleanly on Testcontainers `postgres:18` per the integration test suite.

5. **Translation-key parity stayed clean.**
   Every new UI string in E2.S2/S3/S4 landed in both `de.json` and `en.json` simultaneously. The patch pass also added new keys (`scopeHeader`, `sourcePickerPlaceholder`) to both locales. No drift.

6. **Refresh discipline (`refreshKey + useEffect`) was applied without coaching.**
   The duplicates page handler in E2.S4 bumped `refreshKey` after every mutation instead of chaining duplicate fetches in click handlers. Matches `docs/07_dos_donts.md` item 13 and the Epic 1 retro reinforcement.

## 3. What didn't work

1. **Action A5 (per-story review for E2.S3) created policy tension.**
   Epic 1 retro action A5 said: "Per-story `bmad-code-review` for E2.S3 (Safe Member Merge) — high-risk, sensitive workflow. Do not bundle." The E2.S3 story file Task 0 marks A5 as `[x]` done. But the project's saved workflow policy (`feedback_bmad_workflow.md`) says: "skip per-story bmad-code-review; bundle review + retrospective at epic boundary". Concrete outcome: no separate per-story review ran for E2.S3; the merge service's seven Critical/High findings (concurrent-merge race, Keycloak SaveChanges ordering, EmailRecipient/EventRegistration pre-dedupe gaps, FK SetNull, hard-reject both-Keycloak, Rejected ExpenseClaim orphan) all surfaced at the epic boundary. The hybrid workflow correctly absorbed them, but A5 was not actually applied.

2. **Cross-aggregate symmetric-pre-dedupe was missed.**
   `MemberMergeService` correctly pre-deduped `MemberSegmentAssignment` on `(MemberId, SegmentId)` before the `ExecuteUpdate` rewrite, but skipped the same pattern for `EmailRecipient` on `(CampaignId, MemberId)` and `EventRegistration` on `(EventId, MemberId)`. The original dev acknowledged the omission in a code comment for `EventRegistration` ("no unique constraint on (MemberId, EventId) -- simple rewrite") — which is the smell. A2 (Symmetric-Guard Checklist) covered field-level symmetry but not aggregate-level symmetry inside a transactional move.

3. **Test data didn't exercise hostile inputs.**
   The LIKE-wildcard injection (P1) survived all 17 `MemberRepositoryDuplicateTests` + the `DuplicateMatcherTests` because no test email contained `_`, `%`, or `\`. The matcher tests covered diacritics, case-folding, +tag aliases — all the friendly normalisations — but no test exercised "what happens when the user-supplied email contains LIKE pattern characters that would be interpreted as wildcards." Adversarial test data is a gap.

4. **EF default delete-behavior (SetNull on self-FK) wasn't questioned.**
   The E2.S3 spec said "self-referencing foreign key to `Members(Id)` (`ON DELETE SET NULL`). No data backfill." The dev followed the spec literally. The boundary review caught that `SET NULL` resurrects merged sources when their target is hard-deleted — a data-integrity hole the spec author didn't see. Pre-implementation review of FK behavior on each new self/cross reference would catch this earlier.

5. **A4 (mid-epic patch-collection checkpoint) was self-reporting and toothless.**
   E2.S2 Task 7 marked A4 done by counting open `[Patch]` items between E2.S1 and E2.S2 (count was ≤3, so "continue per bundled plan"). But at that point no story-level review had run, so the count was trivially zero. A4 as written doesn't actually trigger an early-warning — it only fires if mid-epic reviews already produced patches, which they don't in the bundled workflow. Action item shape was wrong.

## 4. Cross-story patterns

- **Concurrency / TOCTOU windows appeared in 3 of 4 stories.** E2.S3 had the blocker→transaction race (P5) and the Keycloak unique-index SaveChanges race (P6). E2.S4 had the dismissal unique-pair insert race (P4). E2.S2 had the frontend debounced-fetch race (P8 — orphaned in-flight `fetch`es without `AbortController`). No story spec required transactional row-locking explicitly. Pattern → A6.

- **Privacy-respecting DTO discipline held.** Despite multiple endpoints (`/duplicates`, `/duplicate-groups`, 409 conflict response) all returning `DuplicateCandidateDto` or `ExistingMemberId`, the response shapes never leaked Phone/Address/Keycloak. The reflection-based test in `FindMemberDuplicatesQueryHandlerTests` is the lock-in for this.

- **Backend-authoritative-with-fail-open-UI pattern.** The frontend `findMemberDuplicates` swallows network errors and lets the form submit; backend's normalized-email guard catches Exact duplicates with a 409. This is the documented design (E2.S2 AC-8). The boundary review flagged it as a UX concern (defer F5), not a security concern, because the backend remains the source of truth.

- **"Cascade-on-action" UX pattern crystallised in D6.** N>2 group dismissal originally recorded only one pair per click, making the group reappear with the other C(N,2)-1 pairs still live. The fix iterates all canonical pairs in a single user action via `Promise.all`. This pattern (single user gesture → all-or-nothing back-end fan-out) is the right shape for any future "operate on a group" UI in Epic 5 (automation rules) or Epic 6 (cost-center reassignment).

- **18 patches across 9 files in a single pass was viable but not comfortable.** The dotnet build + test cycle ran 4 times during patch application; each run takes ~2.5 minutes for the full suite. Cumulative ~10 minutes of test-cycle time. Future passes >15 patches should probably split into 2 commits/test-cycles for sanity.

## 6. Hybrid-workflow re-assessment

**Recommendation: keep the hybrid workflow. Drop A5-style "per-story review for high-risk stories" — it duplicates the boundary review with no marginal value when the boundary review uses three adversarial layers.**

- Epic 2 had one Critical-risk story (E2.S3 Safe Member Merge — touches finance, privacy, auth-critical, >300 LoC). The boundary review with Blind Hunter + Edge Case Hunter + Acceptance Auditor found 7 Critical/High issues in that story, all fixable in the same patch pass.
- The cost of a separate per-story review is: scheduling friction, doubled report-reading, story-level vs. cross-story context split, fragmented patch-tracking. Marginal value when the boundary review is multi-layer is near zero.
- A4 (mid-epic checkpoint) is also marginal under the bundled workflow — no patches accumulate mid-epic to count.

**New escalation trigger (replacing A4/A5):**
- If at any point during epic dev work, the developer feels the boundary patch-cost will exceed ~15 items (rough proxy: more than 1 day of inline patch work), pause and run a `bmad-code-review` against the current accumulated diff as a mid-epic course-correction. This is developer-judgment-driven, not threshold-counted.
