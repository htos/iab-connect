## 3. What didn't go well — patterns

- **Round-1 fixes created round-2 problems — twice.** *(systemic pattern, not blame)*
  - **Round-1 [Review][Patch] in `ModuleEnabledEndpointFilter`** made the public filter fail-OPEN on infra failure. The companion `ModuleAuthorizationHandler` (authenticated surface) was NOT touched → silent asymmetry. Round-2 surfaced this as **DN-3** and had to make the explicit fail-graceful 403 decision. → **A24**
  - **Round-1 [Review][Patch] in `hasAuthSession`** matched cookie name-shape (chunked / `__Host-` / authjs variants). It was already a half-measure: cookie shape ≠ JWT validity. A forged `authjs.session-token=anything` cookie defeated the neutral-UX rewrite. Round-2 surfaced this as **DN-1** and full JWT validation via `getToken({ req, secret })` replaced the regex. → **A24**
- **Patch demotion mid-application.** **P-S4-4 (Sidebar submenu pre-filter test)** was added to the patch list during triage. When applying it, the inline arrow `isModuleEnabled` filter inside `Sidebar.tsx` turned out to be untestable without exporting/refactoring `navItems` — an inline arrow with module-private state. Demoted to defer mid-flight. Should have been recognised as untestable during triage, not during application. → **A25**
- **30 patch findings across 2 review rounds for 5 stories.** Diminishing returns or natural depth? Round-1: 13 patches (mostly Auditor-flagged ACs + first adversarial pass). Round-2: 16+1 patches (mostly Edge Case Hunter & Blind Hunter depth + 3 PARTIAL test-gaps the Auditor surfaced). The pattern suggests: round-1 catches the *story-spec* misses; round-2 catches the *implementation-style + security-posture* misses. A reasonable cost for a security-relevant epic — but worth naming explicitly. → **A26**
- **Dev-API restart left as user action.** Stopped PID 76640 to free DLL locks for `dotnet test`. The api process needs manual restart (`cd backend/src/IabConnect.Api ; dotnet run`). Not documented as a terminal step; surfaced only inline in the final summary. → **A27**
- **One round-2 IDE-Roslyn analyzer false-positive trap.** The `using IabConnect.Domain.Common;` was first added (correct), then the analyzer flagged it as "unnecessary", then removed (incorrect — Domain.Common is NOT in the implicit-usings set for this project), then re-added after a compile-error. Roslyn IDE hints fought with the actual compiler. Burned ~3 tool calls. The lesson: trust compiler errors, not IDE hints. → not action-itemed (situational).
