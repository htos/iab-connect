## 3. Challenges & lessons

- **The story's "S1 stays green unchanged" premise was partly wrong (S2 DEC-1c).** The specs mock `@/lib/auth` with only `useAuth`, so introducing `useApiClient` did NOT transparently survive ‚Äî the transport mocks HAD to be adapted. This is the established E23 members pattern, but the S2 story text mis-stated it. ‚Üí action **A88**.
- **`useApiClient` still cannot express structured error bodies (A84 recurs).** `VolunteerSelfSignupSection` reads `ApiResult.errorBody.errorCode` (4 volunteer conflict branches) which `useApiClient`'s `{data,error,status}` drops ‚Äî exactly the A84 class from E23. It was correctly RETAINED on `@/lib/services/events`, the one deliberate gap to fully emptying `events.ts`. ‚Üí action **A89** (the recurring cost of the lean client contract is now worth a deliberate enhancement decision).
- **Disabled-query surface drift (E24-CR1, Med).** Migrating a page whose god-page early-returned without clearing `loading` to a TanStack `enabled:false` query changes the out-of-role surface (skeleton-forever ‚Üí `permissionDenied` alert). Fees adopted the new surface; volunteers preserved the old quirk ‚Üí inconsistency. ‚Üí action **A90**.
- **Per-domain api-file split vs the "extend events-api.ts" guardrail.** To make parallel authoring conflict-free, each sub-domain got its own `event-<domain>-api.ts` (members-style) sharing one `eventsKeys`/`EVENTS_BASE`. Intent (no duplicated keys/types) honoured; the literal "one file" wording was not. Worth encoding as the accepted pattern. ‚Üí action **A91**.
- **Slice surface has dead exports (E24-CR5).** Five registration hooks + signUp/withdraw mutations + `getMyWaitlistPosition` were built but the components call the api fns imperatively. Parallel agents over-produced the "complete" hook surface. Prune-or-adopt is a cleanup item.

## 5. Action items (promote to project-context.md, A88‚ÄìA91)

- **A88** ‚Äî When a story claims "characterization specs stay green unchanged" across a transport migration, treat the **auth/transport mock surface** (`useApiClient` added to the `@/lib/auth` mock; `fetch`/service-spy ‚Üí `apiClient` spy; endpoint-string assertions) as the EXPECTED licensed change (A79), not a behaviour regression. Pre-state the spec-adaptation in the story so it isn't mistaken for breakage. [root: E24-S2 DEC-1c]
- **A89** (escalates A84) ‚Äî the `useApiClient` `{data,error,status}` contract has now blocked TWO migrations (E23 member 409-synthesis; E24 volunteer self-signup `errorCode`). Make a deliberate decision: either (a) extend the client to surface `errorBody`/ProblemDetails extras, or (b) standardise the "keep on slice-local raw `fetch` + documented exception" pattern. Until then, every endpoint reading body extras stays a documented client exception. [root: E24-S3 DEC-1c / E24-CR6]
- **A90** (refines A79) ‚Äî when migrating a god-page whose load effect early-returns WITHOUT clearing `loading`, the TanStack `enabled:false` equivalent changes the out-of-role/unauth surface (skeleton-forever ‚Üí alert/null). Decide the target surface ONCE per epic and apply it consistently across sibling sub-pages (don't let one page adopt the new surface while another preserves the quirk), and add a test for the chosen surface. [root: E24-CR1]
- **A91** (confirmed approach) ‚Äî a multi-sub-domain slice MAY split its api into per-domain files (`<domain>-api.ts`) that share one `*Keys` factory + base-URL const, mirroring the members slice. This satisfies the "no duplicated keys/types" guardrail AND enables conflict-free parallel multi-agent authoring. Prefer it over one monolithic api file when ‚â•3 sub-domains migrate in parallel.
