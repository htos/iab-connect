## What could improve / what bit us

- **The "already-ships" pattern has now recurred three epics running** (E14-S2 security headers, E17-S1/S2 Serilog/CorrelationId, E18-S3/S4 banner/feedback). Each time, create-story had to discover at refresh that the AC-described feature was already built. This is now predictable enough to be a standard spike step, not a per-epic surprise.
- **A45 binary-reachability bit again** (RUNBOOK `| jq`), exactly as Epic-15 P2 (`aws s3`). The in-story A42/A45 reread caught file/anchor issues but not the "is this binary in the actual runtime image" check for a tool I don't exercise — the boundary review caught it.
- **Repo-wide frontend quality gates are no longer usable as a pass/fail gate.** `npm run lint` reports pre-existing errors in unrelated files and `npm run format:check` reports ~2543 style-drifted files. I had to scope eslint/prettier to changed files to get a meaningful signal. This is accumulating tech debt that will eventually hide a real regression.

## Action items (promoted to project-context.md)

- **A56 — Existing-implementation spike for feature/UI stories at create-story time (generalises A52).** A52 says "verify a SCP-referenced *endpoint* exists at refresh time". Extend it: when a story's AC describes a UI component, a config file, a CI workflow, or any named artifact (`BetaBanner.tsx`, `BetaBanner`, a `.github/` file, an `appsettings.*` key), create-story MUST grep the codebase for an existing implementation BEFORE authoring the story as a build-story. If found, author it as **verification + regression-coverage** (per the E14-S2 / E17-S1 / E18-S3 precedent) and record the AC-literal-vs-shipped divergences in Refresh Notes. Pattern: three epics in a row shipped features the AC described as net-new.
- **A57 — Runtime-image binary inventory check for documented commands (refines A45).** When a doc prescribes a shell command for the `api`/`web`/`keycloak` runtime image, the A42/A45 reread MUST diff the invoked binaries against the actual image inventory. Canonical `api` image inventory (from `backend/Dockerfile`): `curl`, `postgresql-client-17` (`pg_dump`/`pg_restore`/`psql`), plus base `.NET`/shell. **NOT present:** `jq`, `openssl`-GCM-recipe, `aws`, `mc`. A command using an absent binary must either move to the operator's workstation (and say so) or be marked `[!]`. Pattern: Epic-15 P2 (`aws s3`) + Epic-18 P1 (`jq`).
- **A58 — Frontend quality gates run on changed files, not repo-wide, until a format/lint cleanup story lands.** `npm run lint` + `npm run format:check` carry pre-existing repo-wide drift (2 lint errors in unrelated components, ~2543 prettier-drifted files) that makes the repo-wide commands useless as a per-story gate. Until a dedicated cleanup story runs `prettier --write .` + fixes the lint errors, dev-story MUST gate on `eslint <changed-files>` + `prettier --check <changed-files>` and record that the repo-wide drift is pre-existing. **Follow-up candidate:** a standalone "frontend format/lint cleanup" chore (out of Beta critical path; suggest Wave-10 or a quiet slot).
